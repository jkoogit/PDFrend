# PDFrend 백엔드 아키텍처 최종 요약

## 🎯 핵심 설계 원칙

PDFrend 백엔드는 다음 **4가지 핵심 원칙**을 기반으로 설계되었습니다:

| 원칙 | 설명 | 기대 효과 |
| :---: | :--- | :--- |
| **도메인 중심** | 비즈니스 로직이 기술 구현을 주도 | 요구사항 변화에 빠른 대응 |
| **독립성** | 각 도메인이 독립적으로 작동 | 팀 병렬 개발, MSA 전환 용이 |
| **명확성** | 계층과 책임이 명확하게 분리 | 코드 이해도 향상, 유지보수 용이 |
| **확장성** | 새로운 기능 추가가 용이한 구조 | 기술 부채 최소화, 성능 최적화 가능 |

## 📐 아키텍처 구조

### 계층 구조

```
Presentation Layer (React 19 + tRPC)
    ↓ Type-safe RPC
Application Layer (유스케이스, 트랜잭션 관리)
    ↓ 도메인 이벤트 (느슨한 결합)
Domain Layer (8개 도메인, 비즈니스 로직)
    ├─ User Domain (사용자 관리)
    ├─ Document Domain (문서 관리)
    ├─ Version Domain (버전 관리)
    ├─ Annotation Domain (주석 관리)
    ├─ Sharing Domain (공유 & 권한)
    ├─ Sync Domain (실시간 동기화)
    ├─ Storage Domain (파일 저장소)
    └─ Audit Domain (감시 로깅)
    ↓ 저장소 인터페이스 (의존성 역전)
Infrastructure Layer (기술 구현)
    ├─ PostgreSQL + Drizzle ORM
    ├─ S3/NAS 저장소
    ├─ OAuth 인증
    └─ 외부 서비스
```

### 도메인 간 상호작용

```
Document Domain
    ↓ [DocumentUploadedEvent 발행]
Event Publisher
    ├─→ Audit Domain (감시 로그)
    ├─→ User Domain (저장소 할당량)
    └─→ Notification Domain (알림)
```

```
dev → stg → prd (단방향 승격 모델)
```

## ⚙️ 배포 환경별 설정

### 설정 분류

| 분류 | 저장 위치 | 관리 방식 | 예시 |
| :---: | :--- | :--- | :--- |
| **공통 설정** | `config/common.ts` | Git 관리 | API 버전, 페이지 크기 |
| **환경별 설정** | `config/dev.ts`, `stg.ts`, `prd.ts` | Git 관리 | DB 호스트, 로그 레벨 |
| **민감정보** | 환경 변수 | 배포 시스템 주입 | DB 암호, API 키 |
| **로컬 개발** | `.env.local` | Git 제외 | 로컬 DB, 개발 API 키 |

### 환경 변수 주입

**Cloudtype (클라우드):**
- 대시보드에서 환경 변수 설정
- 배포 시 자동으로 주입

**Docker (로컬/NAS):**
- `docker-compose.yml`에서 환경 변수 정의
- `.env.docker` 파일로 민감정보 관리

**로컬 개발:**
- `.env.local` 파일로 로컬 설정
- Git에서 제외 (.gitignore)

## 🚀 개발 시작 가이드

### 1단계: 저장소 클론 및 환경 설정

```bash
# 저장소 클론
git clone https://github.com/your-org/PDFrend_Project.git
cd PDFrend_Project/src/backend

# 의존성 설치
pnpm install

# 환경 변수 설정
cp .env.example .env.local
# .env.local 편집 (로컬 값 설정)
```

### 2단계: PostgreSQL 설정

```bash
# Docker로 PostgreSQL 실행
docker run --name pdfrend-dev-postgres \
  -e POSTGRES_USER=pdfrend_dev \
  -e POSTGRES_PASSWORD=local_dev_password \
  -e POSTGRES_DB=pdfrendd \
  -p 5432:5432 \
  -d postgres:16-alpine

# 데이터베이스 마이그레이션
pnpm drizzle-kit push
```

### 3단계: 서버 시작

```bash
# 개발 서버 시작
pnpm dev

# 예상 출력:
# Server running on http://localhost:3000/
```

### 4단계: 테스트 및 검증

```bash
# 테스트 실행
pnpm test

# 코드 품질 확인
pnpm check

# tRPC 엔드포인트 테스트
curl http://localhost:3000/api/trpc/auth.me
```

## 🔄 개발 워크플로우

### 기능 개발 절차

```
1. 이슈 생성 (#001)
   ↓
2. 브랜치 생성 (feat/#001-feature-name)
   ↓
3. 코드 작성 (도메인 → 애플리케이션 → 라우터)
   ↓
4. 테스트 작성 (유닛 테스트 95% 이상)
   ↓
5. 커밋 (git config user.name "jkman")
   ↓
6. Pull Request (dev 브랜치로)
   ↓
7. 코드 리뷰 및 머지
   ↓
8. dev → stg → prd (자동 승격)
```

### Git 커밋 규칙

```
<type>: <subject> (#<issue-number>)

<body>

<footer>
```

**타입:** feat, fix, refactor, test, docs, chore

**예시:**
```
feat: User 도메인 구현 (#001)

- User 엔티티 및 값 객체 정의
- UserDomainService 구현
- UserRepositoryImpl 구현
- 유닛 테스트 작성 (95% 커버리지)

Closes #001
```

4. **테스트를 먼저 작성하세요**: TDD로 품질 보증
