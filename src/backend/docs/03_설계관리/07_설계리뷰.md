# [설계 검토] PDFrend 주요 이슈 분석 및 대안 제안

## 1. 온라인 실시간 작업 및 권한 리스크 (검토A 관련)

### 1.1 실시간 반영의 리스크
온라인 실시간 반영은 사용자 경험 측면에서 매우 훌륭하지만, 다음과 같은 기술적 리스크가 존재합니다.
*   **네트워크 지연(Latency)**: 작업자 A와 B가 아주 짧은 간격으로 동일한 주석을 수정할 경우, 서버에 도착하는 순서에 따라 데이터가 덮어씌워질 수 있습니다.
*   **충돌(Conflict)**: 작업자 A가 주석을 수정하는 동안 작업자 B가 해당 주석을 삭제하면, A의 수정 요청이 실패하거나 논리적 오류가 발생할 수 있습니다.

### 1.2 권한 기반 충돌 방지 전략
사용자께서 제안하신 **'자기 주석만 편집'** 방식은 가장 확실한 충돌 방지책입니다.
*   **일반 작업자**: 본인이 생성한 주석(UUID 기반 소유권 확인)에 대해서만 수정/삭제 권한 부여. 타인의 주석은 '읽기'만 가능.
*   **문서 소유자(Master)**: 모든 주석에 대한 편집 권한 보유. 
    *   **리스크**: 소유자가 타인의 주석을 수정하는 순간, 해당 주석의 원작자가 오프라인 상태였다면 나중에 동기화 시 충돌이 발생할 수 있습니다.
    *   **해결책**: 소유자가 타인의 주석을 수정할 경우, 해당 주석의 `modified_by` 필드를 소유자로 업데이트하고 원작자에게 알림을 보냅니다.

## 2. 버전 생성 시 타 작업자 처리 방안 (검토B 관련)

동시 작업 중 한 명(주로 소유자)이 '버전 생성(Checkpoint)'을 수행할 때의 처리 방안입니다.

### 제안: 방안 C의 고도화 (버전 업데이트 및 작업 계속)
가장 매끄러운 사용자 경험을 위해 **방안 C**를 추천하며, 다음과 같은 상세 로직을 제안합니다.

1.  **버전 생성 트리거**: 작업자 A가 'v1.0'에서 'v2.0'으로 버전 생성을 클릭.
2.  **서버 액션**: 현재까지의 모든 실시간 주석 데이터를 'v1.0'의 최종 상태로 고정(Snapshot)하고, 새로운 'v2.0' 작업 환경을 생성.
3.  **타 작업자(B, C) 알림**: WebSocket을 통해 "소유자에 의해 v2.0 버전이 생성되었습니다. 현재 작업은 v2.0에 자동으로 저장됩니다." 메시지 표시.
4.  **연속성 유지**: 작업자 B와 C는 화면 새로고침 없이 계속 작업하며, 이후 발생하는 모든 저장 요청은 서버에서 자동으로 'v2.0'으로 라벨링되어 저장됨.

## 3. 오프라인 동기화 및 메타데이터 관리

### 3.1 메타정보 기반 판단 (사용자 제안 반영)
파일 해시 대신 **문서 내 메타정보(Metadata)**를 활용하는 방식에 전적으로 동의합니다.
*   **관리 항목**: `Document_ID`, `Current_Version`, `Last_Modified_By`, `Last_Timestamp`.
*   **판단 로직**:
    *   **동일성**: `Document_ID`가 일치하는가?
    *   **최신성**: 로컬의 `Last_Timestamp`가 서버의 `Last_Timestamp`보다 앞서는가?
    *   **수정여부**: 로컬 IndexedDB에 서버로 전송되지 않은 'Pending' 상태의 주석 데이터가 있는가?

### 3.2 변경점 선택 방식 (Selective Merge)
오프라인 작업 후 온라인 전환 시, 단순히 파일을 덮어쓰는 것이 아니라 **주석 단위로 변경점을 리스트업**하여 사용자가 선택하게 합니다.
*   예: "오프라인에서 추가한 5개의 형광펜 주석과 2개의 텍스트 주석이 있습니다. 서버에 반영하시겠습니까?"
*   사용자는 체크박스로 반영할 항목만 선택하여 'Merge' 수행.

## 4. DB 구조 고도화 (RDBMS + JSONB)

### 4.1 테이블 설계 전략
*   **Annotations 테이블**:
    *   `id` (UUID): 주석 고유 식별자
    *   `doc_id` (FK): 문서 ID
    *   `page_num` (Int): 페이지 번호
    *   `user_id` (FK): 작성자 ID
    *   `data` (JSONB): 좌표, 색상, 텍스트 내용, 타입(Highlight, Text 등)
    *   `version` (Float): 해당 주석이 속한 버전
*   **장점**: `data->>'type' = 'Highlight'`와 같은 쿼리로 특정 타입의 주석만 초고속 검색 가능.
