# 테스트 전략 및 계획

## 📌 개요

PDFrend 백엔드의 테스트 전략은 **품질 보증**, **회귀 방지**, **신뢰성 확보**를 목표로 합니다. 단위 테스트부터 E2E 테스트까지 계층화된 테스트 체계를 구축하여 각 단계에서 결함을 조기에 발견하고 해결합니다.

---

## 🎯 테스트 목표

| 목표 | 설명 | 기대 효과 |
| :---: | :--- | :--- |
| **품질 보증** | 각 기능이 요구사항을 만족하는지 검증 | 사용자 만족도 향상 |
| **회귀 방지** | 기존 기능이 손상되지 않았는지 확인 | 개발 속도 향상 |
| **신뢰성 확보** | 시스템이 안정적으로 작동하는지 검증 | 운영 안정성 보장 |
| **문서화** | 코드의 동작을 테스트로 문서화 | 유지보수 용이 |
| **리팩토링 안전성** | 코드 개선 시 기능 손상 방지 | 기술 부채 감소 |

## 🏗️ 테스트 피라미드

```
        ┌─────────────┐
        │   E2E Test  │  (5-10%)
        │   (느림)    │  사용자 여정 테스트
        └─────────────┘
           ▲       ▲
          /         \
         /           \
        ┌─────────────┐
        │Integration  │  (20-30%)
        │   Test      │  도메인 간 상호작용
        │  (중간속도) │
        └─────────────┘
           ▲       ▲
          /         \
         /           \
        ┌─────────────┐
        │  Unit Test  │  (60-70%)
        │  (빠름)     │  개별 함수/클래스
        └─────────────┘
```

**테스트 비율:**
- **단위 테스트:** 60-70% (빠르고 격리된 테스트)
- **통합 테스트:** 20-30% (모듈 간 상호작용)
- **E2E 테스트:** 5-10% (사용자 시나리오)

## 🧪 단위 테스트 (Unit Tests)

### 목적

단위 테스트는 **개별 함수, 클래스, 메서드**가 예상대로 작동하는지 검증합니다. 외부 의존성(DB, API)을 Mock으로 대체하여 빠르고 격리된 테스트를 수행합니다.

### 테스트 대상

단위 테스트는 다음 계층의 코드를 주로 대상으로 합니다:

**1. 도메인 레이어 (Domain Layer)**

도메인 레이어는 비즈니스 로직을 포함하므로 가장 중요한 테스트 대상입니다.

```typescript
// User 엔티티 테스트 예시
describe('User Entity', () => {
  it('should create a user with valid data', () => {
    const user = User.create({
      openId: 'user-123',
      name: 'John Doe',
      email: 'john@example.com',
    });

    expect(user.id).toBeDefined();
    expect(user.openId).toBe('user-123');
    expect(user.storageUsed).toBe(0);
  });

  it('should throw error when uploading file exceeds quota', () => {
    const user = User.create({
      openId: 'user-123',
      name: 'John Doe',
      storageQuota: 1000, // 1KB
      storageUsed: 900,   // 900B 사용 중
    });

    expect(() => {
      user.canUploadFile(200); // 200B 파일 업로드 시도
    }).toThrow('Storage quota exceeded');
  });
});
```

**2. 애플리케이션 레이어 (Application Layer)**

애플리케이션 서비스의 비즈니스 로직을 테스트합니다.

```typescript
// UserApplicationService 테스트 예시
describe('UserApplicationService', () => {
  let service: UserApplicationService;
  let mockRepository: MockUserRepository;

  beforeEach(() => {
    mockRepository = new MockUserRepository();
    service = new UserApplicationService(mockRepository);
  });

  it('should register a new user', async () => {
    const result = await service.registerUser({
      openId: 'user-123',
      name: 'John Doe',
      email: 'john@example.com',
    });

    expect(result.id).toBeDefined();
    expect(mockRepository.save).toHaveBeenCalled();
  });
});
```

**3. 값 객체 (Value Objects)**

불변 객체의 생성, 검증, 비교 로직을 테스트합니다.

```typescript
// StorageInfo 값 객체 테스트 예시
describe('StorageInfo Value Object', () => {
  it('should calculate remaining quota', () => {
    const storage = new StorageInfo(1000, 300); // 1000 total, 300 used

    expect(storage.remaining).toBe(700);
    expect(storage.usagePercentage).toBe(30);
  });

  it('should throw error when used exceeds quota', () => {
    expect(() => {
      new StorageInfo(1000, 1500); // 1500 > 1000
    }).toThrow('Used storage cannot exceed quota');
  });
});
```

### 테스트 작성 원칙

**1. 테스트 격리 (Test Isolation)**

각 테스트는 독립적으로 실행되어야 하며, 다른 테스트의 결과에 영향을 받지 않아야 합니다.

```typescript
// ❌ 나쁜 예: 테스트 간 의존성
let user: User;

describe('User', () => {
  it('should create user', () => {
    user = User.create({ /* ... */ });
  });

  it('should update user', () => {
    // 이전 테스트에 의존
    user.updateName('New Name');
    expect(user.name).toBe('New Name');
  });
});

// ✅ 좋은 예: 각 테스트가 독립적
describe('User', () => {
  it('should create user', () => {
    const user = User.create({ /* ... */ });
    expect(user.id).toBeDefined();
  });

  it('should update user', () => {
    const user = User.create({ /* ... */ });
    user.updateName('New Name');
    expect(user.name).toBe('New Name');
  });
});
```

**2. Mock 사용 (Mocking External Dependencies)**

외부 의존성(DB, API)을 Mock으로 대체하여 테스트를 빠르고 격리되게 만듭니다.

```typescript
// Mock Repository 예시
class MockUserRepository implements UserRepository {
  private users: Map<number, User> = new Map();

  async findById(id: number): Promise<User | null> {
    return this.users.get(id) || null;
  }

  async save(user: User): Promise<void> {
    this.users.set(user.id, user);
  }
}

// 테스트에서 사용
describe('UserApplicationService', () => {
  it('should find user by id', async () => {
    const mockRepo = new MockUserRepository();
    const service = new UserApplicationService(mockRepo);

    const user = User.create({ /* ... */ });
    await mockRepo.save(user);

    const found = await service.getUserById(user.id);
    expect(found).toEqual(user);
  });
});
```

**3. AAA 패턴 (Arrange-Act-Assert)**

테스트를 명확하게 구조화합니다.

```typescript
describe('User', () => {
  it('should calculate storage usage percentage', () => {
    // Arrange: 테스트 데이터 준비
    const user = User.create({
      storageQuota: 1000,
      storageUsed: 300,
    });

    // Act: 테스트할 동작 수행
    const percentage = user.getStorageUsagePercentage();

    // Assert: 결과 검증
    expect(percentage).toBe(30);
  });
});
```

**4. 명확한 테스트 이름**

테스트 이름으로 의도가 명확하게 드러나야 합니다.

```typescript
// ❌ 나쁜 예: 모호한 이름
it('should work', () => { /* ... */ });

// ✅ 좋은 예: 명확한 이름

it('should calculate remaining storage correctly', () => { /* ... */ });
it('should not allow file upload if quota is exceeded', () => { /* ... */ });
```

## 🔗 통합 테스트 (Integration Tests)

### 목적

통합 테스트는 **여러 모듈, 계층, 도메인 간의 상호작용**을 검증합니다. 실제 데이터베이스를 사용하거나 테스트 데이터베이스를 별도로 구성하여 진행합니다.

### 테스트 대상

**1. 도메인 간 상호작용**

여러 도메인이 함께 작동하는 시나리오를 테스트합니다.

```typescript
// User와 Document 도메인 간 상호작용 테스트
describe('User and Document Integration', () => {
  let userService: UserApplicationService;
  let documentService: DocumentApplicationService;
  let testDb: TestDatabase;

  beforeEach(async () => {
    testDb = new TestDatabase();
    await testDb.setup();
    userService = new UserApplicationService(new UserRepositoryImpl(testDb));
    documentService = new DocumentApplicationService(new DocumentRepositoryImpl(testDb));
  });

  afterEach(async () => {
    await testDb.teardown();
  });

  it('should update user storage when document is uploaded', async () => {
    // 1. 사용자 생성
    const user = await userService.registerUser({
      openId: 'user-123',
      name: 'John Doe',
    });

    // 2. 문서 업로드
    const document = await documentService.uploadDocument({
      userId: user.id,
      fileName: 'test.pdf',
      fileSize: 5000000, // 5MB
    });

    // 3. 사용자 저장소 사용량 확인
    const updatedUser = await userService.getUserProfile(user.id);
    expect(updatedUser.storageUsed).toBe(5000000);
  });
});
```

**2. 데이터베이스 통합 테스트**

Repository 구현과 실제 데이터베이스 상호작용을 테스트합니다.

```typescript
// UserRepositoryImpl 통합 테스트
describe('UserRepositoryImpl Integration', () => {
  let repository: UserRepositoryImpl;
  let testDb: TestDatabase;

  beforeEach(async () => {
    testDb = new TestDatabase();
    await testDb.setup();
    repository = new UserRepositoryImpl(testDb);
  });

  afterEach(async () => {
    await testDb.teardown();
  });

  it('should save and retrieve user', async () => {
    const user = User.create({
      openId: 'user-123',
      name: 'John Doe',
      email: 'john@example.com',
    });

    await repository.save(user);
    const retrieved = await repository.findById(user.id);

    expect(retrieved).toEqual(user);
  });

  it('should find user by openId', async () => {
    const user = User.create({
      openId: 'user-123',
      name: 'John Doe',
    });

    await repository.save(user);
    const retrieved = await repository.findByOpenId('user-123');

    expect(retrieved?.id).toBe(user.id);
  });

  it('should update user', async () => {
    const user = User.create({
      openId: 'user-123',
      name: 'John Doe',
    });

    await repository.save(user);
    user.updateName('Jane Doe');
    await repository.update(user);

    const retrieved = await repository.findById(user.id);
    expect(retrieved?.name).toBe('Jane Doe');
  });
});
```

**3. tRPC 라우터 통합 테스트**

tRPC 라우터와 전체 요청-응답 흐름을 테스트합니다.

```typescript
// tRPC 라우터 통합 테스트
describe('User tRPC Router Integration', () => {
  let caller: ReturnType<typeof appRouter.createCaller>;
  let testDb: TestDatabase;

  beforeEach(async () => {
    testDb = new TestDatabase();
    await testDb.setup();
    caller = appRouter.createCaller({
      db: testDb,
      user: { id: 1, role: 'user' }, // Mock 사용자
    });
  });

  afterEach(async () => {
    await testDb.teardown();
  });

  it('should get user profile', async () => {
    const profile = await caller.user.profile();

    expect(profile.id).toBe(1);
    expect(profile.name).toBeDefined();
  });

  it('should update user profile', async () => {
    const updated = await caller.user.updateProfile({
      name: 'New Name',
      email: 'new@example.com',
    });

    expect(updated.name).toBe('New Name');
    expect(updated.email).toBe('new@example.com');
  });

  it('should throw error for unauthorized access', async () => {
    const unAuthCaller = appRouter.createCaller({
      db: testDb,
      user: null, // 인증되지 않은 사용자
    });

    await expect(unAuthCaller.user.profile()).rejects.toThrow('UNAUTHORIZED');
  });
});
```

**4. 도메인 이벤트 통합 테스트**

도메인 이벤트 발행 및 처리를 테스트합니다.

```typescript
// 도메인 이벤트 통합 테스트
describe('Domain Events Integration', () => {
  let eventBus: EventBus;
  let userService: UserApplicationService;
  let auditService: AuditApplicationService;

  beforeEach(() => {
    eventBus = new InMemoryEventBus();
    userService = new UserApplicationService(
      new UserRepositoryImpl(testDb),
      eventBus
    );
    auditService = new AuditApplicationService(
      new AuditRepositoryImpl(testDb),
      eventBus
    );

    // 이벤트 핸들러 등록
    eventBus.subscribe(UserCreatedEvent, (event) => {
      auditService.logUserCreated(event);
    });
  });

  it('should publish and handle UserCreatedEvent', async () => {
    const user = await userService.registerUser({
      openId: 'user-123',
      name: 'John Doe',
    });

    // 이벤트가 처리될 때까지 대기
    await new Promise(resolve => setTimeout(resolve, 100));

    // 감시 로그 확인
    const logs = await auditService.getLogsForUser(user.id);
    expect(logs).toContainEqual(
      expect.objectContaining({
        action: 'USER_CREATED',
        userId: user.id,
      })
    );
  });
});
```

## 🌐 E2E 테스트 (End-to-End Tests)

### 목적

E2E 테스트는 **사용자 관점에서 전체 시스템**이 올바르게 작동하는지 검증합니다. 프론트엔드부터 백엔드, 데이터베이스까지 전체 스택을 테스트합니다.

### 테스트 시나리오

**1. 사용자 가입 시나리오**

```typescript
describe('User Registration E2E', () => {
  it('should complete user registration flow', async () => {
    // 1. 로그인 페이지 접근
    await page.goto('http://localhost:3000/login');

    // 2. OAuth 로그인
    await page.click('[data-testid="oauth-login-button"]');
    // OAuth 팝업 처리...

    // 3. 프로필 설정
    await page.fill('[data-testid="name-input"]', 'John Doe');
    await page.fill('[data-testid="email-input"]', 'john@example.com');
    await page.click('[data-testid="save-profile-button"]');

    // 4. 대시보드 확인
    await expect(page).toHaveURL('http://localhost:3000/dashboard');
    await expect(page.locator('[data-testid="user-name"]')).toContainText('John Doe');
  });
});
```

**2. 문서 업로드 시나리오**

```typescript
describe('Document Upload E2E', () => {
  it('should upload document and verify storage usage', async () => {
    // 1. 대시보드 접근
    await page.goto('http://localhost:3000/dashboard');

    // 2. 초기 저장소 사용량 확인
    const initialStorage = await page.locator('[data-testid="storage-used"]').textContent();

    // 3. 파일 업로드
    const fileInput = await page.locator('[data-testid="file-input"]');
    await fileInput.setInputFiles('test.pdf');
    await page.click('[data-testid="upload-button"]');

    // 4. 업로드 완료 대기
    await expect(page.locator('[data-testid="upload-success"]')).toBeVisible();

    // 5. 저장소 사용량 업데이트 확인
    const updatedStorage = await page.locator('[data-testid="storage-used"]').textContent();
    expect(updatedStorage).not.toBe(initialStorage);

    // 6. 문서 목록에서 확인
    await expect(page.locator('text=test.pdf')).toBeVisible();
  });
});
```

pnpm test --reporter=verbose

## 🎓 테스트 작성 팁

**1. 테스트 우선 개발 (TDD)**

기능 구현 전에 테스트를 먼저 작성합니다.

```typescript
// 1. 테스트 작성 (실패)
it('should calculate storage usage percentage', () => {
  const user = User.create({ storageQuota: 1000, storageUsed: 300 });
  expect(user.getStorageUsagePercentage()).toBe(30);
});

// 2. 최소 구현 (통과)
class User {
  getStorageUsagePercentage(): number {
    return (this.storageUsed / this.storageQuota) * 100;
  }
}

// 3. 리팩토링 (개선)
class User {
  private calculateUsagePercentage(): number {
    if (this.storageQuota === 0) return 0;
    return (this.storageUsed / this.storageQuota) * 100;
  }
}
```

**2. 테스트 유지보수**

테스트 코드도 프로덕션 코드처럼 유지보수해야 합니다.

- 중복 제거 (DRY 원칙)
- 명확한 이름
- 적절한 주석
- 정기적인 리팩토링

**3. 테스트 실패 분석**

테스트가 실패하면 원인을 명확히 파악합니다.

- 코드 버그인가?
- 테스트 가정이 잘못되었나?
- 요구사항 변경인가?
