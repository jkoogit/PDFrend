# 260513_02_심층_#3_CRUD비즈니스로직구현분석

## 1. 분석 개요
이슈 #3에서는 PDFrend의 핵심 도메인인 User와 Document에 대한 CRUD(Create, Read, Update, Delete) 로직을 tRPC와 Drizzle ORM을 사용하여 구현하였습니다. 본 보고서는 구현된 코드의 구조, 주요 결정 사항 및 기술적 특징을 분석합니다.

## 2. 주요 구현 내용

### 2.1. tRPC 라우터 구조
`apps/backend/src/api/trpc/router.ts` 파일에 모든 비즈니스 로직을 집약하였습니다. tRPC의 `procedure`를 사용하여 입력 값 검증(Zod)과 비즈니스 로직을 하나의 단위로 관리합니다.

### 2.2. Drizzle ORM 활용
*   **Insert/Update/Delete**: `.returning()` 메서드를 사용하여 작업 결과를 즉시 반환받아 API 응답으로 활용합니다.
*   **Query**: `eq`, `and` 등 Drizzle의 쿼리 빌더를 사용하여 타입 안정성이 보장된 SQL을 생성합니다.

### 2.3. 보안 및 권한 제어
Document CRUD 작업 시, 단순히 `id`만으로 접근하는 것이 아니라 `userId`를 함께 조건절(`and(eq(documents.id, id), eq(documents.userId, userId))`)에 포함하여 타인의 문서에 접근하거나 수정하는 것을 방지하였습니다.

## 3. 기술적 결정 사항

| 결정 사항 | 이유 |
|---|---|
| **Zod 기반 입력 검증** | tRPC와의 완벽한 통합 및 런타임/컴파일 타임 타입 안정성 확보 |
| **returning() 사용** | 추가적인 SELECT 쿼리 없이 생성/수정된 데이터를 클라이언트에 즉시 전달하여 효율성 증대 |
| **Flat Config (ESLint)** | ESLint v9 이상의 최신 표준을 준수하여 프로젝트 유지보수성 향상 |

## 4. 학습 포인트
*   **tRPC Error Handling**: `TRPCError`를 사용하여 클라이언트에 명확한 HTTP 상태 코드와 메시지를 전달하는 방법을 익혔습니다.
*   **Drizzle ORM Transaction**: 현재는 단일 쿼리 위주이나, 향후 복합적인 로직에서는 `db.transaction` 도입이 필요함을 인지하였습니다.

## 5. 결론 및 향후 과제
이슈 #3 구현을 통해 백엔드의 핵심 API 체계가 마련되었습니다. 다음 단계로는 실제 파일 업로드 로직 연동 및 프론트엔드에서의 API 호출 테스트가 필요합니다.
