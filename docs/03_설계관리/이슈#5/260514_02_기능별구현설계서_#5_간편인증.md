# 260514_02_기능별구현설계서_#5_간편인증

## 0. 문서 성격 및 적용 범위
- **문서 성격: 목표안(최종안)**
- 본 문서는 이슈 #5 인증 체계의 최종 목표 구조(기본 id/pw + 부가 인증수단: 이메일/간편인증)를 정의한다.
- 비용/일정 제약으로 인한 단계적 릴리즈 범위(예: 이메일 우선 적용)는 별도 실행계획/이슈 문서에서 관리하며, 본 문서의 목표 구조를 변경하지 않는다.

## 1. 문서 목적
이슈 #5 간편인증(이메일 포함) 구현을 기능 단위로 분해하여, 개발 순서/입출력/예외/검증 게이트를 명확히 정의한다.

## 2. 공통 설계 원칙
- 인증수단 우선순위: **없음** (사용자 선택 로그인)
- 계정 주체 식별자: 내부 `회원 UUID` 고정, 인증수단(이메일/간편인증)은 연결 정보로 관리
- 민감정보 비노출: access token/secret/provider secret은 로그/문서 원문 금지
- 오류 표준화: 인증 오류는 도메인 오류코드 체계로 통일(`AUTH_*`)
- 기존 로컬 인증 호환: local/login 흐름과 충돌하지 않도록 계정연계 정책 고정

## 3. 기능별 설계

### F1. 인증 시작 (Auth Start: 이메일/간편인증 공통 게이트)
- 목표: 클라이언트 요청 시 provider 인증 URL로 안전하게 리다이렉트한다.
- 입력: `provider`, `redirectUri`, `state`(CSRF 방어), 필요 scope
- 출력: provider authorize URL 또는 서버 리다이렉트 응답
- 실패/예외:
  1) 미지원 provider
  2) redirectUri 불일치
  3) state 생성/저장 실패
- 검증 포인트:
  - state가 세션/저장소에 정상 보관되는가
  - provider별 scope가 정책과 일치하는가

### F2. 인증 검증 처리 (Auth Verify)
- 목표: 인증수단별 검증 결과를 획득하고 내부 회원계정(UUID)과 안전하게 매핑한다.
- 입력: `provider`, `authRequest/state`(간편인증 계열), `email verification token`(이메일 계열)
- 처리:
  1) state 검증
  2) 채널별 검증(예: 간편인증 검증 결과 확인, 이메일 토큰 검증)
  3) 인증수단 식별정보 조회(`providerId/subKey`, `email`, `email_verified`)
  4) 내부 회원계정 매핑/생성/연결
- 출력: 내부 인증 토큰 발급 준비 컨텍스트 + 인증수단 링크 상태
- 실패/예외:
  - 인증요청 만료/위변조
  - state mismatch
  - provider profile 필수값 누락

### F3. 계정 매핑/연결 정책
- 목표: **회원계정(UUID)** 과 복수 인증수단(이메일/간편인증) 매핑을 안전하게 운영한다.
- 서비스 구체화 반영:
  1) 회원가입 시 이메일 인증 + 간편인증(전화 기반 포함)을 지원한다.
  2) 알림 발송은 "인증 완료된 연락처"로만 수행한다(계획).
  3) 로그인 후 사용자정보 화면에서 이메일 재인증, 전화번호 재인증(간편인증 갱신)을 지원한다.
  4) 인증수단 정보는 복수 관리(다대일: 여러 인증수단 -> 하나의 회원계정)한다.
- 용어 정의:
  - 회원계정: 가입 시 생성되는 내부 고객 계정(고유키: `user.uuid`)
  - 인증수단: 이메일 인증, 간편인증(provider/subKey, 전화번호 기반 가능)
  - 인증갱신: 기존 회원계정에 연결된 인증수단의 재검증 또는 키 갱신
- 핵심 정책:
  1) `users.email`은 유니크로 관리한다.
  2) 간편인증 키는 `(provider, subKey)` 복합 유니크로 관리한다.
  3) 이메일 인증과 간편인증은 자동 동일인 병합을 하지 않는다.
  4) 이미 등록된 이메일/전화번호/간편인증키로 가입 시도 시 신규 생성 대신 "등록된 회원" 안내 후 로그인 유도한다.
  5) 인증수단 추가/변경은 로그인 세션 + 재인증 성공 시에만 반영한다.
  6) 전화번호 변경/재인증 시 기존 번호를 즉시 삭제하지 않고 이력 상태로 보관하되, 알림 대상은 최신 인증 완료 번호 1건으로 제한한다.
- 데이터 무결성:
  - `users(uuid)` PK, `users(email)` UNIQUE
  - `auth_identity(provider, sub_key)` UNIQUE
  - `user_auth_link(user_uuid, identity_id, status, verified_at)` UNIQUE
  - 연락처 알림 발송 대상은 `verified=true` + `is_primary=true`(최종 인증번호) 조건으로 제한
- 예상 충돌 및 검토 피드백:
  - **충돌 가능성은 낮지만 0은 아님**(주요 원인: 연락처 재할당/번호 재사용, provider subKey 변경, 수동 데이터 보정).
  - 따라서 "가입시 충돌 회피"만으로 충분하지 않고, 인증갱신 시점의 재검증/중복체크 트랜잭션이 필요하다.
  - 번호 변경 후에도 구 번호/신 번호가 모두 유효할 수 있으므로, 발송 채널은 `최종 인증번호(primary)`만 사용해야 중복 알림을 방지할 수 있다.
- 실패/예외:
  - 동일 간편인증 키의 다중 계정 연결 시도
  - 인증갱신 중 이미 다른 계정에 선점된 연락처로 변경 시도
  - 전화번호 교체 이후 이전 번호 사용자와 신규 번호 사용자 간 알림 중복 발생
  - 운영 중 수동 보정 데이터와 유니크 제약 충돌

### F4. 토큰 발급/재발급 정책
- 목표: access 토큰 단기, refresh(선택) 장기 정책으로 보안/편의 균형 확보
- 기본안:
  - access token: 짧은 TTL
  - refresh token: 회전(rotating) 전략 검토
- 재발급 정책:
  1) refresh 유효 + 블랙리스트 미등록 -> 재발급 허용
  2) refresh 탈취 의심/재사용 탐지 -> 즉시 폐기 + 재로그인 유도
- 실패/예외:
  - 만료 토큰 재발급 요청
  - 블랙리스트 등록 토큰 요청

### F5. 로그아웃/토큰 무효화
- 목표: 로그아웃 즉시 보호 API 접근 차단
- 처리:
  1) 현재 access(및 refresh 사용 시 refresh 포함) 블랙리스트 등록
  2) TTL 기반 정리 정책 적용
- 저장소:
  - 1차: 메모리 denylist
  - 2차: Redis/DB 영속화(후속)
- 실패/예외:
  - 저장소 장애 시 fallback 처리(최소 경고 로그 + 재시도 정책)

### F6. 권한 연계(Role)
- 목표: 인증 후 인가까지 연결
- 최소 정책:
  - user 기본 role 부여
  - 보호 리소스는 소유자 우선 접근
- 확장 정책:
  - admin/editor/user 세분화 및 정책표 기반 검증

## 4. 기능별 테스트 전략
- 단위 테스트(필수)
  - state 생성/검증
  - 계정 매핑 분기
  - 토큰 발급/검증/블랙리스트 체크
- 통합 테스트(선택)
  - provider mock 기반 callback~token 발급 E2E 흐름
- 인수 테스트(선택)
  - 실제 배포 URL 기준 주요 사용자 시나리오

## 5. 구현 순서 제안
1. F3(계정 주체/인증수단 매핑 규칙) 확정 및 충돌 트랜잭션 고정
2. F1/F2(인증 시작/검증) 구현: 이메일 + 간편인증 공통 게이트 우선
3. F4/F5(토큰/로그아웃) 정책 고정
4. F6(Role) 최소 연계
5. 단위 테스트 우선 완료 후 통합/인수 범위 확정

## 5-1. 단계안(릴리즈) 주석
- **현재 단계안(비용 제약):** 릴리즈 순서는 이메일 인증 우선 적용 가능
- **최종 목표안 유지:** 간편인증 연동은 제외가 아닌 후속 스프린트 편성 대상으로 관리
- 원칙: 단계안은 구현 순서만 조정하며, 본 문서의 목표 구조(F1~F6)는 유지한다.
