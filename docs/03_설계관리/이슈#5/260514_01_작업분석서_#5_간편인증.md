# 260514_01_작업분석서_#5_간편인증

## 1. 목적 (G)
이슈 #5(간편인증 중심 인증체계) 착수를 위해, 정책/설계/일정/의사결정 문서를 기준으로 작업대상·선행조건·리스크·완료기준을 확정한다.
본 문서는 새 채팅/세션에서도 즉시 재개 가능한 기준 문서로 사용한다.

## 2. 기준 문서 분석 (B)

### 2.1 Git 전략/작업 표준 흐름
- 브랜치 전략은 `feat(or feature) -> dev -> stg -> main` 순차 승격 모델을 적용한다.
- 작업 시작은 이슈 등록, 작업브랜치 생성, 작업대상 분석, 실행계획, DoD 수립 순서(GBRD 1단계)로 진행한다.
- PR 머지 전 정적검사(`tsc --noEmit`, ESLint) 및 배포 스모크 확인이 필수다.

### 2.2 개발 가이드/기술 정책
- 인증 구현은 Node.js + Express + tRPC + TypeScript + Drizzle 환경에서 수행한다.
- 민감 정보는 문서에 원문 기록 금지, 환경변수 관리대장에 항목 기준으로만 관리한다.
- 간편인증/이메일 인증 가이드를 참조하여 provider별 검증/토큰 처리 표준을 맞춘다.

### 2.3 기획/설계/의사결정/일정 기준
- 로드맵상 인증 고도화는 코어 백엔드 안정화 구간의 핵심 항목이다.
- ADR-003(간편인증/이메일 병행 방향)과 기존 인증 구현(#4) 결과를 연속선상에서 확장해야 한다.
- 기존 잔여업무(Role, 재발급, 블랙리스트 영속화, 자동화 테스트)는 #5 설계에 반드시 연결한다.

## 4. 현재 배포/환경/정책 고정값

### 4.1 현재 배포 상태 (새 채팅 공유 항목)
- dev/stg/main 승급 완료 상태를 기준으로 이슈 #5를 시작한다.
- 서비스 URL 및 마지막 정상 응답 시간은 배포 스모크 단계에서 최신값으로 갱신한다.

### 4.2 환경변수 현황
- `DATABASE_URL`: 환경별 매핑 상태 점검 필요(누락/오매핑 여부 확인).
- 인증 시크릿: access/refresh(or session) 관련 키의 환경별 설정 및 회전 정책 점검 필요.

### 4.3 정책 고정값
- 비용 제약으로 간편인증 고도화는 단계 적용, 이메일 인증 우선 릴리즈
- 이슈 번호: `#5`
- 작업 브랜치 명명: `feature/#5-social-auth`
- 머지 전략: Squash/Rebase는 PR 생성 시 팀 규칙에 맞춰 확정하고, 필수 체크(타입/린트/테스트/스모크) 통과를 머지 게이트로 적용

## 5. 미해결 잔여업무 연계 목록

2. 토큰 재발급
3. 블랙리스트 영속화
4. 자동화 테스트

## 6. 리스크 및 대응
1. Provider별 스펙 차이로 인한 인증 플로우 불일치
   - 대응: 공통 어댑터 계층/표준 에러코드 정의
2. 회원계정-인증수단(이메일/간편인증) 매핑 충돌
   - 시나리오 A: 등록된 이메일/전화번호로 회원가입(간편인증 포함) 재시도하는 경우
   - 시나리오 B: 로그인 후 이메일/전화번호 재인증 중, 대상 연락처가 다른 계정에 이미 연결된 경우
   - 시나리오 C: provider `subKey` 변경 또는 번호 재사용으로 기존 매핑이 달라지는 경우
   - 시나리오 D: "등록된 회원입니다. 로그인하시겠습니까." 안내 이후 로그인 전환 과정에서 세션 혼선이 발생하는 경우
   - 대응 원칙:
     1) 회원 주체는 `user.uuid`로 고정, 인증수단은 복수 링크로 관리
     2) `users.email`, `(provider, subKey)` 유니크 제약으로 중복 가입 차단
     3) 신규가입 충돌 시 계정 생성 대신 로그인 유도 메시지 및 안전한 전환 플로우 제공
     4) 인증수단 갱신은 로그인 세션 + 재인증 성공 + 원자적 중복검사 트랜잭션으로 처리
     5) 알림은 `verified=true` + `is_primary=true` 최종 인증 연락처로만 발송
   - 대응 산출물: 매핑 규칙표, 충돌 시나리오 테스트 목록, 인증갱신 트랜잭션 명세
3. 토큰/세션 정책 혼재
   - 대응: access/refresh 수명, 재발급 조건, 로그아웃 무효화 정책을 단일 표로 고정
4. 테스트 공백
   - 대응: 단위 테스트는 필수, 통합/인수 테스트는 위험도 및 배포 범위에 따라 선택 적용

### 7.2 실패 시나리오 (각 3개 세부 기준)
