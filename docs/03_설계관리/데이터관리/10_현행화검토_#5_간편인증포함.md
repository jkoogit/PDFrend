# 10. 현행화 검토서 #5 (간편인증 포함)

## 1. 목적
지금까지 작성된 이슈 #5 문서(기획/설계/데이터/기술)의 정합성을 재검토하고, PR 전 기준 현행 상태를 확정한다.

## 2. 검토 범위
- 기획: 인증 전략 방향성(이메일 채널 분리, 간편인증 로그인)
- 설계: 동일인 식별/계정연결 정책, 테이블/도메인 규칙
- 기술: 카카오/네이버/구글 API 연동 절차 및 운영 체크리스트

## 3. 검토 결론 요약
1. **간편인증 도입 가능**: 휴대폰 본인인증 없이도 카카오/네이버/구글 연동 가능
2. **동일인 자동 병합 금지**: `provider_cd + provider_key_val`를 1차 식별자로 고정
3. **이메일 정책 정정 완료**: 이메일 인증은 로그인 필수조건이 아니라 협업/공지 채널
4. **데이터 표준 정리 완료**:
   - 테이블ID: `TB_AUTH_USER`, `TB_AUTH_IDENTITY`, `TB_AUTH_LINK`, `TB_AUTH_EVENT`
   - 메타컬럼: `reg_system`, `reg_user_id`, `reg_dtm`, `mod_system`, `mod_user_id`, `mod_dtm`
   - 코드 도메인: `cd_10`, 상태 도메인: `status_cd`

## 4. 현행화 반영 체크
- [x] 데이터 사전(단어/용어/도메인/공통코드) 정합성 반영
- [x] 인증 테이블 컬럼명 용어 기반 정리
- [x] 모델 개요 문서에 PK/FK/UK 반영
- [x] 간편인증 API 연동 매뉴얼 Step-by-Step 상세화
- [x] 이메일/간편인증 독립 정책 반영

## 5. PR 전 최종 권고
1. 이번 PR은 **문서 기준선(Baseline) 확정 PR**로 머지
2. 다음 작업은 구현 브랜치에서 아래 순서로 진행
   - OAuth 콜백/토큰 교환 구현
   - `provider_key_val` 기반 로그인/연결 구현
   - 인증수단 등록/삭제 API + 감사이력(auth_events)
