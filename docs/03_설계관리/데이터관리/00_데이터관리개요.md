# 00. 데이터관리개요

## 1. 목적
이 문서는 데이터관리 정책(사전/도메인/코드/명명규칙/메타정보)을 통합 관리한다.

## 2. 핵심 질의 답변
1. PostgreSQL에 UUID 타입이 있나?
   - **있다.** PostgreSQL은 `uuid` 내장 타입을 지원한다.
2. 도메인명 뒤 `_dom` 의미?
   - 사내 표준에서 "도메인 정의값"임을 식별하기 위한 접미사다.
3. `true/false` vs `Y/N` 차이?
   - `boolean(true/false)` 장점: 타입 안정성, 조건식 단순, 인덱스/체크 일관
   - `Y/N` 장점: 레거시/외부연계 친화
   - 권장: DB 저장은 `boolean`, 외부 인터페이스 변환은 어댑터에서 처리
4. 코드값을 `active`처럼 단어로 쓸지 `1000/W001`처럼 코드로 쓸지?
   - 단어형 장점: 가독성 우수
   - 숫자/기호형 장점: 변경 내성, 다국어 분리 용이
   - 권장: 저장은 코드형(예: `AUTH_STATUS_ACTIVE` 또는 `1000`), 표시는 공통코드 사전 라벨 사용
5. text 길이 100/500/1000 가능?
   - PostgreSQL `varchar(n)`에서 가능하다. `varchar(1000)`도 가능.

## 3. 데이터 정책
### 3.1 사전관리
- 단어사전/용어사전/도메인사전/공통코드사전을 표준으로 관리한다.
- 사전 변경 시 테이블 문서와 동시 현행화한다.

### 3.2 명명 규칙
- 컬럼명은 `업무접두 + 용어` 구조를 따른다.
- 컬럼ID는 `용어 + 용어(도메인)`으로 관리한다.
- 하나의 용어에 여러 도메인(예: CNTNV100/CNTNV1000/CNTNBLOB)이 매핑될 수 있다.
- 용어의 마지막 단어는 도메인 성격과 일치해야 한다.
  - 예) `user_id`는 `id` 도메인 적용
  - 예) `link_status`는 `status` 도메인 적용
  - 예) `primary_yn`은 `yn` 도메인 적용
- 사전에 없는 단어 사용 금지(`usr` 금지, `user` 사용).

### 3.3 메타정보 공통 컬럼
- 모든 테이블 공통 6종:
  - `reg_system`, `reg_user`, `reg_dtm`
  - `mod_system`, `mod_user`, `mod_dtm`

### 3.4 타입/도메인 정책
- 식별자: `uuid`, `id_num`, `id`
- 상태코드: `status_cd`
- 여부값: `yn(boolean)`
- 문자열: `text_100`, `text_500`, `text_1000`

### 3.5 코드 정책
- 코드값은 공통코드사전 기준으로 관리한다.
- DB 저장은 코드형, 화면 표시는 코드 라벨 매핑으로 분리한다.


## 5. 상태코드/시스템코드 도메인 필요성 정리

질문: `D0004(상태코드)`, `D0005(시스템코드)`는 공통코드로 관리하니 도메인이 불필요한가?

- 결론: **실무에서는 공통코드 + 도메인을 함께 두는 경우가 일반적**이다.
- 이유:
  1) 공통코드: "허용 값 집합"을 관리 (무슨 코드가 가능한가)
  2) 도메인: "물리 타입/길이/기본 제약"을 관리 (어떻게 저장되는가)
- 즉, 역할이 다르므로 중복이 아니라 **분리 책임**이다.

### 권장 운영안
- 공통코드사전: 코드 의미/라벨/사용처 관리
- 도메인사전: `varchar(10)` 등 저장 규격 관리
- 테이블 설계: 상태/시스템 컬럼은 코드 도메인(`cd_10`)을 사용하고, 값은 공통코드에서 통제

### 예외(도메인 생략 가능)
- 프로젝트 규모가 작고 코드 컬럼이 1~2개뿐이며, 타입/길이 표준화 이점이 거의 없을 때
- 단, 중장기적으로는 재사용성과 변경통제 측면에서 도메인 분리가 유리
