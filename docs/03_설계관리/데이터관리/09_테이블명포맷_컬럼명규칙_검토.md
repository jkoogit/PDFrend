# 09. 테이블명 포맷 및 컬럼명 규칙 검토

## 1) 테이블명 포맷 비교 (사용자 테이블 기준)
| 후보 | 예시 | 장점 | 단점 | 권장도 |
| :--- | :--- | :--- | :--- | :---: |
| 순번형 | `TB_AUTH_001` | 단순, 정렬 용이 | 의미 해석 어려움, 변경 시 문맥 약함 | △ |
| 의미형 | `TB_AUTH_USER` | 의미 명확, 운영/쿼리 가독성 높음 | 순서/버전 정보 없음 | ◎ |
| 혼합형 | `TB_AUTH001_USER` | 영역+순번+의미 동시 표현 | 길고 복잡, 오타 가능성 증가 | ○ |

### 일반적인 구성
- `TB_<도메인>_<엔터티>` 형태가 가장 보편적
- 예: `TB_AUTH_USER`, `TB_AUTH_IDENTITY`, `TB_AUTH_EVENT`

### 결론
- **실무 권장: 의미형(`TB_AUTH_USER`)**
- 관리대장에 별도 `테이블ID(TB_AUTH_001)`를 유지하면 순번 관리도 가능

## 2) `email_cntn` vs `email_addr`
- 권장: **`email_addr`**
  - 이유: 의미가 명확(주소), `cntn`은 범용이라 과도하게 추상적
- 도메인 권장: `addr_v500` (varchar(500))

## 3) `user_name_cntn` vs `user_nm`
- 권장: **`user_nm`**
  - 이유: 관례적이고 간결, 명시성 충분
- 도메인 권장: `nm_v50`, `nm_v100`, `nm_v500`

## 4) `user_status_cd` vs `status_cd` (테이블명에 USER 포함 시)
- `TB_AUTH_USER` 테이블 내부에서는 **`status_cd` 권장**
  - 이유: 중복 접두 제거로 간결
  - 단, JOIN/뷰에서 혼동이 우려되면 alias로 보완
- 범용 규칙
  - 테이블 문맥이 충분하면 짧게(`status_cd`)
  - 문맥이 약한 통합뷰/ETL 산출 컬럼은 접두 포함(`user_status_cd`)

## 5) `provider_sub_key_cntn` vs `provider_sub_key_val`
- 권장: **`provider_sub_key_val`**
  - 이유: 키의 "값"이라는 성격이 명확
- 값 도메인 세트 권장
  - 문자: `val_v10`, `val_v100`, `val_v500`, `val_v1000`
  - 숫자: `val_n5`, `val_n10`, `val_n50`, `val_n100`
  - 소수: `val_n5_2`, `val_n10_2`

## 6) 메타정보 컬럼 표준
- `reg_system`, `reg_user_id`, `reg_dtm`, `mod_system`, `mod_user_id`, `mod_dtm`
