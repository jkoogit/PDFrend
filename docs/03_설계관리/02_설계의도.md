# 09. 아키텍처 설계 의도 및 설명

## 목적

이 문서는 PDFrend 백엔드 아키텍처의 **설계 의도(Design Rationale)**를 상세히 설명합니다. 각 아키텍처 결정의 배경, 고려사항, 그리고 기대 효과를 명확히 하여, 향후 개발자들이 설계 철학을 이해하고 일관되게 유지할 수 있도록 합니다.

---

## 1. 아키텍처 개요

### 1.1 핵심 설계 철학

PDFrend 백엔드 아키텍처는 다음 **4가지 핵심 원칙**을 기반으로 설계되었습니다:

| 원칙 | 설명 | 기대 효과 |
| :---: | :--- | :--- |
| **도메인 중심** | 비즈니스 로직이 기술 구현을 주도 | 요구사항 변화에 빠른 대응 |
| **독립성** | 각 도메인이 독립적으로 작동 | 팀 병렬 개발, MSA 전환 용이 |
| **명확성** | 계층과 책임이 명확하게 분리 | 코드 이해도 향상, 유지보수 용이 |
| **확장성** | 새로운 기능 추가가 용이한 구조 | 기술 부채 최소화, 성능 최적화 가능 |

## 2. 도메인 주도 설계(DDD) 적용 의도

### 2.1 왜 DDD를 선택했는가?

**PDFrend의 특성:**
- 복잡한 버전 관리 및 변경 추적 로직
- 실시간 협업 및 충돌 해결 알고리즘
- 다양한 공유 권한 및 접근 제어
- 향후 기능 확장 예상 (OCR, AI 분석 등)

**DDD의 이점:**
1. **비즈니스 로직 중심**: 기술이 아닌 비즈니스 개념으로 설계
2. **명확한 경계**: 각 도메인의 책임이 명확하게 분리
3. **언어 통일**: 개발자, 기획자, 운영팀이 같은 언어 사용
4. **테스트 용이**: 도메인 로직이 기술에 독립적

### 2.2 8개 도메인 분해의 의도

| 도메인 | 분해 이유 | 기대 효과 |
| :---: | :--- | :--- |
| **User** | 인증, 권한, 저장소 할당량 | 사용자 관리 독립화 |
| **Document** | 문서 생명주기 관리 | 핵심 도메인 격리 |
| **Version** | 버전 관리 및 이력 추적 | 변경 이력 관리 독립화 |
| **Annotation** | 주석 및 협업 | 협업 기능 모듈화 |
| **Sharing** | 공유 및 권한 | 접근 제어 독립화 |
| **Sync** | 실시간 동기화 | 동기화 로직 격리 |
| **Storage** | 파일 저장소 | 저장소 추상화 (S3/NAS) |
| **Audit** | 감시 및 로깅 | 감사 추적 독립화 |

**분해 기준:**
- 각 도메인은 **하나의 비즈니스 개념**만 담당
- 도메인 간 의존성은 **도메인 이벤트**로 관리
- 각 도메인은 **독립적으로 테스트** 가능
- 향후 **마이크로서비스로 분리** 가능

### 2.3 계층 분리의 의도

**Domain Layer (비즈니스 로직):**

```typescript
// 도메인 로직은 기술에 독립적
class Document {
  canBeModifiedBy(userId: number): boolean {
    return this.ownerId === userId;
  }

  archive(): void {
    if (this.status !== 'active') {
      throw new Error('Only active documents can be archived');
    }
    this.status = 'archived';
  }
}
```

**의도:**
- 비즈니스 규칙이 명확하게 표현됨
- 데이터베이스, 프레임워크 변경에 영향 없음
- 도메인 로직을 쉽게 테스트할 수 있음

**Application Layer (유스케이스):**

```typescript
// 애플리케이션 서비스는 도메인 서비스를 조율
class DocumentApplicationService {
  async uploadNewVersion(
    documentId: UUID,
    userId: number,
    file: Buffer
  ): Promise<void> {
    // 1. 문서 소유권 확인 (도메인 로직)
    const document = await this.documentRepository.findById(documentId);
    if (!document.canBeModifiedBy(userId)) {
      throw new Error('Permission denied');
    }

    // 2. 파일 저장 (인프라)
    const fileReference = await this.storageService.upload(file);

    // 3. 버전 생성 (도메인 로직)
    const version = Version.create(documentId, userId, fileReference);
    await this.versionRepository.save(version);

    // 4. 도메인 이벤트 발행 (느슨한 결합)
    await this.eventPublisher.publish(
      new DocumentVersionUploadedEvent(documentId, version.id, userId)
    );
  }
}
```

**의도:**
- 유스케이스가 명확하게 표현됨
- 도메인 로직과 기술 구현이 분리됨
- 트랜잭션 관리가 중앙화됨

**Infrastructure Layer (기술 구현):**

```typescript
// 인프라는 도메인 인터페이스를 구현
class DocumentRepositoryImpl implements DocumentRepository {
  async findById(id: UUID): Promise<Document | null> {
    const row = await db.query(
      'SELECT * FROM documents WHERE id = $1',
      [id]
    );
    if (!row) return null;
    return this.toDomain(row);
  }

  private toDomain(row: any): Document {
    return new Document(
      row.id,
      row.owner_id,
      new DocumentMetadata(
        row.title,
        row.file_name,
        row.file_hash,
        row.file_size,
        row.mime_type
      ),
      row.current_version_id,
      row.status,
      row.created_at,
      row.updated_at
    );
  }
}
```

**의도:**
- 기술 구현이 도메인에 영향을 주지 않음
- 데이터베이스, ORM 변경이 도메인에 영향 없음
- 테스트 시 Mock 구현으로 쉽게 대체 가능

## 3. 다중 데이터베이스 전략의 의도

### 3.1 왜 dev, stg, prd를 분리했는가?

**개발 환경 (pdfrendd):**
- 개발자가 자유롭게 스키마 변경 가능
- 테스트 데이터로 오염되지 않음
- 다른 개발자에게 영향 없음

**검증 환경 (pdfrends):**
- 실제 데이터와 유사한 구조로 QA 검증
- 마스킹된 데이터로 개인정보 보호
- 프로덕션 배포 전 최종 확인

**운영 환경 (pdfrend):**
- 실제 사용자 데이터 보호
- 엄격한 접근 제어
- 자동 백업 및 복제

**기대 효과:**
- 개발 중 실수가 운영 환경에 영향을 주지 않음
- 데이터 품질 보증
- 규정 준수 (개인정보보호법 등)

### 3.2 단방향 승격 모델의 의도

```
dev → stg → prd (한 방향만)
```

**vs 양방향 동기화:**

| 특성 | 단방향 승격 | 양방향 동기화 |
| :---: | :--- | :--- |
| **데이터 무결성** | ✅ 보장 | ❌ 위험 |
| **이력 추적** | ✅ 명확 | ❌ 복잡 |
| **롤백** | ✅ 간단 | ❌ 어려움 |
| **운영 안정성** | ✅ 높음 | ❌ 낮음 |

**의도:**
- 각 환경의 데이터 무결성 보장
- 배포 이력이 명확하게 기록됨
- 문제 발생 시 빠른 롤백 가능

## 5. 도메인 이벤트 패턴의 의도

### 5.1 왜 도메인 이벤트를 사용하는가?

**도메인 이벤트 없이 (강한 결합):**

```typescript
// ❌ 도메인 간 강한 결합
class DocumentService {
  async uploadDocument(file: Buffer): Promise<void> {
    // 1. 문서 저장
    const document = await this.documentRepository.save(file);

    // 2. 감시 로그 기록 (Document가 Audit을 알아야 함)
    await this.auditService.log('DOCUMENT_UPLOADED', document.id);

    // 3. 알림 발송 (Document가 Notification을 알아야 함)
    await this.notificationService.notify(document.ownerId, 'Document uploaded');

    // 4. 저장소 할당량 업데이트 (Document가 User를 알아야 함)
    await this.userService.updateStorageUsed(document.ownerId, file.size);
  }
}
```

**문제점:**
- Document 도메인이 Audit, Notification, User 도메인에 의존
- 새로운 기능 추가 시마다 Document 서비스 수정 필요
- 도메인 간 순환 의존성 위험

**도메인 이벤트 사용 (느슨한 결합):**

```typescript
// ✅ 도메인 이벤트로 느슨한 결합
class DocumentService {
  async uploadDocument(file: Buffer): Promise<void> {
    // 1. 문서 저장
    const document = await this.documentRepository.save(file);

    // 2. 도메인 이벤트 발행 (다른 도메인은 몰라도 됨)
    await this.eventPublisher.publish(
      new DocumentUploadedEvent(document.id, document.ownerId, file.size)
    );
  }
}

// 다른 도메인에서 이벤트 구독
class AuditEventHandler {
  async handle(event: DocumentUploadedEvent): Promise<void> {
    await this.auditService.log('DOCUMENT_UPLOADED', event.documentId);
  }
}

class NotificationEventHandler {
  async handle(event: DocumentUploadedEvent): Promise<void> {
    await this.notificationService.notify(
      event.userId,
      'Document uploaded'
    );
  }
}

class UserEventHandler {
  async handle(event: DocumentUploadedEvent): Promise<void> {
    await this.userService.updateStorageUsed(event.userId, event.fileSize);
  }
}
```

**기대 효과:**
- 도메인 간 의존성 제거
- 새로운 기능 추가가 기존 도메인에 영향 없음
- 향후 마이크로서비스 전환 시 메시지 큐로 쉽게 전환 가능

### 5.2 도메인 이벤트 흐름

```
Document Domain
    ↓
    [DocumentUploadedEvent 발행]
    ↓
Event Publisher (메모리 기반, 향후 RabbitMQ로 전환)
    ├─→ Audit Event Handler (감시 로그 기록)
    ├─→ Notification Event Handler (알림 발송)
    └─→ User Event Handler (저장소 할당량 업데이트)
```

## 6. tRPC 선택의 의도

### 6.1 왜 tRPC를 선택했는가?

**REST API vs tRPC:**

| 특성 | REST | tRPC |
| :---: | :--- | :--- |
| **타입 안전성** | ❌ 수동 | ✅ 자동 |
| **API 문서** | ❌ 별도 작성 | ✅ 자동 생성 |
| **클라이언트 생성** | ❌ 수동 | ✅ 자동 |
| **에러 처리** | ❌ 수동 | ✅ 자동 |
| **성능** | ✅ 우수 | ✅ 우수 |

**tRPC의 이점:**
1. **End-to-End 타입 안전성**: 프론트엔드에서 백엔드 타입을 자동으로 추론
2. **개발 속도**: API 문서 작성 불필요
3. **에러 감소**: 타입 미스매치를 컴파일 시점에 감지
4. **DX 향상**: IDE 자동완성으로 개발 경험 향상

### 6.2 tRPC 라우터 구조

```typescript
// server/routers.ts
export const appRouter = t.router({
  // User 도메인
  user: t.router({
    me: publicProcedure.query(async ({ ctx }) => {
      // 현재 사용자 정보 반환
    }),
    profile: protectedProcedure.query(async ({ ctx }) => {
      // 사용자 프로필 반환
    }),
  }),

  // Document 도메인
  document: t.router({
    list: protectedProcedure.query(async ({ ctx }) => {
      // 문서 목록 반환
    }),
    create: protectedProcedure
      .input(z.object({ title: z.string() }))
      .mutation(async ({ ctx, input }) => {
        // 문서 생성
      }),
  }),

  // ... 다른 도메인
});
```

**의도:**
- 각 도메인별로 라우터를 구성
- 입출력 스키마를 명시적으로 정의
- 프론트엔드에서 타입 안전하게 호출 가능

## 7. Drizzle ORM 선택의 의도

### 7.1 왜 Drizzle ORM을 선택했는가?

**ORM 비교:**

| 특성 | TypeORM | Prisma | Drizzle |
| :---: | :--- | :--- | :--- |
| **타입 안전성** | 중간 | 높음 | ✅ 매우 높음 |
| **성능** | 중간 | 중간 | ✅ 우수 |
| **학습곡선** | 가파름 | 완만 | 중간 |
| **유연성** | 낮음 | 낮음 | ✅ 높음 |
| **SQL 제어** | 제한적 | 제한적 | ✅ 완전 제어 |

**Drizzle의 이점:**
1. **TypeScript 우선**: 완벽한 타입 추론
2. **SQL 제어**: 복잡한 쿼리를 직접 작성 가능
3. **마이그레이션**: 스키마 변경을 SQL로 명시적으로 관리
4. **성능**: ORM 오버헤드 최소화

### 7.2 Drizzle 스키마 정의

```typescript
// drizzle/schema.ts
export const users = pgTable('users', {

});

export const documents = pgTable('documents', {

  title: varchar('title', { length: 255 }).notNull(),
  fileName: varchar('file_name', { length: 255 }).notNull(),
  fileHash: varchar('file_hash', { length: 64 }),
  fileSize: bigint('file_size').notNull(),

  createdAt: timestamp('created_at').defaultNow(),

});

export const annotations = pgTable('annotations', {
  id: uuid('id').primaryKey().defaultRandom(),
  documentId: uuid('document_id').notNull().references(() => documents.id),
```

## 8. PostgreSQL JSONB 활용의 의도

### 8.1 왜 JSONB를 사용하는가?

**주석 데이터 저장:**

```typescript
// JSONB로 유연한 주석 데이터 저장
{
  "type": "highlight",
  "page": 5,
  "coordinates": {
    "x": 100,
    "y": 200,
    "width": 300,
    "height": 50
  },
  "color": "#FF0000",
  "text": "Important section"
}
```

**기대 효과:**
- 주석 타입별로 다른 스키마 필요 없음
- 새로운 주석 타입 추가 시 스키마 변경 불필요
- 쿼리 시 JSONB 연산자로 필터링 가능

### 8.2 JSONB 쿼리 예시

```sql
-- 특정 페이지의 주석 조회
SELECT * FROM annotations
WHERE document_id = 'doc-123'
AND (data->>'page')::int = 5;

-- 특정 색상의 주석 조회
SELECT * FROM annotations
WHERE document_id = 'doc-123'
AND data->>'color' = '#FF0000';

-- JSONB 인덱싱으로 성능 최적화
CREATE INDEX idx_annotations_page 
ON annotations ((data->>'page'));
```

### 9.3 마이그레이션 전략

**Phase 1-2 (현재):**
- 모놀리식 아키텍처
- 모든 도메인이 하나의 프로세스에서 실행
- 도메인 이벤트는 메모리 기반 (EventEmitter)

**Phase 3 (준비):**
- 각 도메인별 독립적인 데이터베이스 설계
- 도메인 이벤트를 RabbitMQ로 전환 준비
- API 게이트웨이 설계

**Phase 4 (전환):**
- 도메인별 서비스 분리
- 각 서비스는 독립적인 포트에서 실행
- RabbitMQ를 통한 도메인 이벤트 통신
- 서비스 디스커버리, 로드 밸런싱 구성

**기대 효과:**
- 각 팀이 독립적으로 개발 가능
- 서비스별 독립적인 배포 가능
- 성능 최적화 및 확장성 향상

```typescript
const documents = await db.query('SELECT * FROM documents WHERE owner_id = ?', [userId]);
for (const doc of documents) {

// ✅ 좋은 예: JOIN 사용

```

### 10.3 캐싱 전략

```typescript
// 자주 조회되는 데이터 캐싱
class DocumentService {
  private cache = new Map<string, Document>();

  async getDocument(id: UUID): Promise<Document> {
    // 캐시 확인
    if (this.cache.has(id)) {
      return this.cache.get(id)!;
    }

    // 데이터베이스에서 조회
    const document = await this.documentRepository.findById(id);

    // 캐시에 저장 (5분 TTL)
    this.cache.set(id, document);
    setTimeout(() => this.cache.delete(id), 5 * 60 * 1000);

    return document;
  }
}
```
