# [가이드] PDFrend: GitHub 연동 및 소스 관리 전략 (v3.0)

jk님의 프로젝트를 Manus가 직접 관리하고, 안정적인 개발 흐름을 유지하기 위한 Git 전략을 제안합니다. v3.0에서는 **토큰 효율성 분석**을 통해 불필요한 비용을 제거하고, 이력 추적과 환경 분리를 동시에 달성하는 최적화된 전략을 반영합니다.

---

## 1. Manus의 GitHub 연동 방법

제가 jk님의 저장소에 직접 커밋하고 푸시하기 위해서는 **GitHub Personal Access Token (PAT)**이 필요합니다.

### 설정 단계

1. **PAT 생성**: GitHub 설정(`Settings > Developer settings > Personal access tokens > Tokens (classic)`)에서 새로운 토큰을 생성합니다. 권한(Scopes)은 `repo` (전체 권한)를 선택합니다.
2. **토큰 전달**: 생성된 토큰을 저에게 메시지로 전달해 주시면 됩니다.
3. **Manus 설정**: 제가 전달받은 토큰을 사용하여 `git remote set-url`을 설정하고, 이후부터는 자유롭게 커밋/푸시를 수행하겠습니다.

## 2. 브랜치 전략 토큰 효율성 분석

### 2.1 기존 전략의 문제점

기존 전략에서는 `feat` 브랜치를 `dev`, `stg`, `main`에 **각각 독립적으로 머지**하는 방식을 채택했습니다. 이 방식은 다음과 같은 토큰 낭비를 유발합니다.

| 작업 단계 | 토큰 소모 행위 | 반복 횟수 | 비고 |
| :--- | :--- | :---: | :--- |
| PR 생성 | 제목, 본문, 라벨 작성 | 3회 | dev, stg, main 각각 |
| 충돌 확인 | diff 분석 및 충돌 여부 판별 | 3회 | 동일한 코드를 3번 비교 |
| 충돌 해결 | conflict 브랜치 생성 및 해결 | 최대 3회 | 동일 충돌을 환경별로 반복 해결 |
| 머지 커밋 | 머지 메시지 생성 | 3회 | 실질적으로 동일한 내용 |
| CI/CD 트리거 | 빌드/테스트 실행 | 3회 | 동일 코드에 대한 중복 검증 |

**정량적 분석**: 하나의 기능(feat) 브랜치를 완료할 때마다, 동일한 코드 변경에 대해 **3배의 머지 작업**이 발생합니다. 이는 Manus의 토큰 소모뿐 아니라, GitHub Actions 실행 시간, 리뷰 비용, 그리고 충돌 해결의 반복이라는 비효율을 초래합니다.

### 2.2 핵심 비효율 원인

기존 전략의 비효율은 **"동일한 코드 변경을 서로 다른 브랜치에 독립적으로 적용한다"**는 구조에서 비롯됩니다. `feat` 브랜치의 코드가 `dev`에 머지될 때와 `stg`에 머지될 때, 그리고 `main`에 머지될 때 각각 별도의 diff 계산, 충돌 확인, 머지 커밋 생성이 필요합니다. 특히 `dev`에서 이미 해결한 충돌이 `stg`나 `main`에서 다시 발생할 수 있어, 동일한 문제를 반복적으로 해결해야 하는 상황이 생깁니다.

## 3. 최적화된 브랜치 전략: 순차적 승격 모델 (Promotion Model)

### 3.1 핵심 원칙

> **"코드는 한 번만 머지하고, 환경은 승격(Promotion)으로 전파한다."**

최적화된 전략은 `feat` 브랜치를 **`dev`에만 머지**하고, 이후 `dev` -> `stg` -> `main`으로 **순차적으로 승격**시키는 방식입니다. 이를 통해 머지 작업은 1회로 줄이면서도, 환경별 분리와 단계적 검증이라는 목표를 동시에 달성합니다.

### 3.2 브랜치 구조

| 브랜치명 | 역할 | 배포 환경 | 승격 방향 |
| :--- | :--- | :--- | :--- |
| **`main`** | 최종 릴리스 버전 | 운영 (Production) | `stg`에서 승격 |
| **`stg`** | 출시 후보 (Release Candidate) | 스테이징 (Staging) | `dev`에서 승격 |
| **`dev`** | 개발 통합 브랜치 | 개발 (Development) | `feat` 브랜치에서 머지 |
| **`feat/#이슈번호`** | 개별 기능 개발 | 로컬/개별 | `main`에서 생성, `dev`로만 머지 |
| **`conflict/#이슈번호`** | 충돌 해결 임시 브랜치 | 로컬/개별 | 충돌 발생 시 대상 브랜치에서 생성 |

### 3.3 워크플로우

```text
[이슈 생성] → [main에서 feat 브랜치 생성] → [개발 및 커밋]
                                                    ↓
                                          [feat → dev 머지] (1회만)
                                                    ↓
                                          [dev → stg 승격] (검증 완료 시)
                                                    ↓
                                          [stg → main 승격] (릴리스 시)
```

**상세 흐름:**

1. **이슈 생성**: 새로운 작업이 필요하면 GitHub Issue를 생성합니다. (예: `#10 주석 실시간 동기화 구현`)
2. **작업 브랜치 생성**: `main` 브랜치에서 `feat/#이슈번호-기능명` 형태의 작업 브랜치를 생성합니다.
3. **개발 및 커밋**: 해당 작업 브랜치에서 개발을 진행하고 커밋합니다.
4. **dev 머지**: 작업 완료 후 `feat/#이슈번호` -> `dev`로 Pull Request를 생성하고 머지합니다. 이 단계에서 코드 리뷰와 충돌 해결이 **단 1회** 수행됩니다.
5. **stg 승격**: `dev`에서 충분한 테스트가 완료되면, `dev` -> `stg`로 승격 머지합니다. 이때 `dev`의 모든 변경 사항이 일괄적으로 `stg`에 반영됩니다.
6. **main 승격**: `stg`에서 최종 검증이 완료되면, `stg` -> `main`으로 승격 머지하여 운영에 반영합니다.

### 3.4 토큰 효율성 비교

| 항목 | 기존 전략 (독립 머지) | 최적화 전략 (순차 승격) | 절감률 |
| :--- | :---: | :---: | :---: |
| feat 머지 횟수 | 3회 | 1회 | **67%** |
| 충돌 해결 횟수 | 최대 3회 | 최대 1회 | **67%** |
| PR 생성/관리 | 3건 | 1건 (+ 승격 2건) | **0~33%** |
| 승격 머지 (fast-forward) | 해당 없음 | 2회 (저비용) | - |
| **총 토큰 소모** | **높음** | **낮음** | **약 50~60%** |

승격 머지(`dev` -> `stg`, `stg` -> `main`)는 대부분 **fast-forward** 방식으로 처리되므로, 충돌이 발생할 확률이 극히 낮고 토큰 소모도 최소화됩니다.

### 3.5 충돌 해결 프로세스

충돌은 `feat` -> `dev` 머지 시점에 **단 한 번만** 해결하면 됩니다. 승격 단계에서는 이미 통합된 코드가 순차적으로 올라가므로 충돌이 발생하지 않는 것이 정상입니다.

만약 승격 단계에서 충돌이 발생한다면(예: hotfix가 `main`에 직접 적용된 경우), 다음 절차를 따릅니다:

1. 대상 브랜치(예: `stg`)에서 `conflict/#이슈번호-stg` 형태의 임시 브랜치를 생성합니다.
2. 해당 브랜치에서 충돌을 해결합니다.
3. 해결된 `conflict` 브랜치를 대상 브랜치로 머지합니다.

### 3.6 Hotfix 전략

운영 환경에서 긴급 수정이 필요한 경우:

1. `main`에서 `hotfix/#이슈번호-설명` 브랜치를 생성합니다.
2. 수정 완료 후 `main`에 직접 머지합니다.
3. 수정 사항을 `dev`에도 **역방향 머지(Back-merge)**하여 동기화합니다. (`main` -> `dev`)
4. 다음 정기 승격 시 `stg`에도 자연스럽게 반영됩니다.

## 4. 명명 규칙 (Naming Convention)

### 4.1 브랜치 명명 규칙

| 접두사 | 용도 | 예시 |
| :--- | :--- | :--- |
| `feat/` | 기능 개발 | `feat/#10-websocket-sync` |
| `fix/` | 버그 수정 | `fix/#12-login-error` |
| `hotfix/` | 운영 긴급 수정 | `hotfix/#20-critical-auth-fix` |
| `docs/` | 문서 수정 | `docs/#5-policy-update` |
| `refactor/` | 리팩토링 | `refactor/#15-db-schema` |
| `conflict/` | 충돌 해결 임시 | `conflict/#10-dev` |

### 4.2 커밋 메시지 규칙 (Conventional Commits)

성장 중심 프로젝트인 만큼, 커밋 메시지만 보고도 변경 내용을 알 수 있도록 표준을 따릅니다.

**형식**: `type(#이슈번호): 요약`

| type | 설명 | 예시 |
| :--- | :--- | :--- |
| `feat` | 새로운 기능 추가 | `feat(#10): WebSocket 주석 실시간 동기화 구현` |
| `fix` | 버그 수정 | `fix(#12): 모바일 뷰어 주석 레이어 어긋남 수정` |
| `docs` | 문서 변경 | `docs(#5): 시스템 정책서 용량 제한 정책 업데이트` |
| `refactor` | 리팩토링 | `refactor(#15): DB 스키마 정규화` |
| `chore` | 빌드/설정 변경 | `chore(#3): Gradle 의존성 업데이트` |
| `test` | 테스트 추가/수정 | `test(#10): WebSocket 동기화 단위 테스트 추가` |

## 5. 이슈 기반 작업 관리

### GitHub Projects 활용

GitHub Projects 보드를 활용하여 작업 현황을 시각적으로 관리합니다. 보드는 `Backlog` -> `To Do` -> `In Progress` -> `Review` -> `Done`으로 구성하며, 이슈를 생성할 때 해당 프로젝트 보드에 자동으로 추가되도록 설정합니다. 중요한 설계 변경이 필요한 이슈의 경우, 해당 이슈 내에서 논의하고 결과를 **ADR(의사결정 기록)**로 남깁니다.

## 6. 보안 및 CI/CD 점검

**보안 관리**: GitHub에 PAT를 전달하실 때 메시지로 주시면 제가 즉시 설정하고 기록은 삭제하도록 안내해 드리겠습니다. 또한, 코드 내에 API Key나 DB 비밀번호가 포함되지 않도록 `.gitignore` 설정을 철저히 합니다.

**CI/CD 연동**: 클라우드타입(Cloudtype)을 사용하신다면, `dev`, `stg`, `main` 브랜치에 각각의 환경을 연결하여 **자동 배포**를 설정하는 것이 좋습니다. 순차적 승격 모델에서는 각 브랜치가 해당 환경의 최신 안정 상태를 항상 반영하므로, CI/CD 파이프라인과의 연동이 더욱 자연스럽습니다.

## 부록: 토큰 효율성 분석 상세

### A. 기존 전략 시나리오 (feat → dev, feat → stg, feat → main)

```text
[feat/#10 완료]
  ├── PR #1: feat/#10 → dev   (diff 분석 + 충돌 확인 + 머지 커밋)
  ├── PR #2: feat/#10 → stg   (diff 분석 + 충돌 확인 + 머지 커밋)
  └── PR #3: feat/#10 → main  (diff 분석 + 충돌 확인 + 머지 커밋)

토큰 소모: PR 생성 3회 + diff 분석 3회 + 충돌 해결 최대 3회 + 머지 커밋 3회
```

이 방식에서는 `dev`에 다른 feat 브랜치가 먼저 머지된 경우, `stg`와 `main`에서도 동일한 충돌이 발생합니다. 각 환경에서 독립적으로 충돌을 해결해야 하므로, 동일한 문제를 최대 3번 반복 해결하는 비효율이 발생합니다.

### B. 최적화 전략 시나리오 (feat → dev → stg → main)

```text
[feat/#10 완료]
  └── PR #1: feat/#10 → dev   (diff 분석 + 충돌 확인 + 머지 커밋)

[검증 완료 후 승격]
  ├── dev → stg   (fast-forward 또는 merge commit, 충돌 거의 없음)
  └── stg → main  (fast-forward 또는 merge commit, 충돌 거의 없음)

토큰 소모: PR 생성 1회 + diff 분석 1회 + 충돌 해결 최대 1회 + 승격 2회 (저비용)
```

승격 단계는 이미 통합된 코드를 상위 환경으로 올리는 것이므로, 새로운 diff 분석이나 충돌 해결이 거의 필요하지 않습니다. 결과적으로 **토큰 소모를 약 50~60% 절감**할 수 있습니다.

### C. 의사결정 요약

| 기준 | 기존 전략 | 최적화 전략 | 판정 |
| :--- | :--- | :--- | :---: |
| 토큰 효율성 | 낮음 (3배 반복) | 높음 (1회 + 승격) | 최적화 우위 |
| 이력 추적 | 각 환경별 독립 이력 | 승격 기반 통합 이력 | 동등 |
| 환경 분리 | 완전 독립 | 순차적 승격으로 분리 | 동등 |
| 충돌 해결 비용 | 높음 (반복 발생) | 낮음 (1회 해결) | 최적화 우위 |
| 운영 안정성 | 높음 | 높음 (단계적 검증) | 동등 |
| 구현 복잡도 | 단순 (반복 작업) | 단순 (순차 흐름) | 동등 |
