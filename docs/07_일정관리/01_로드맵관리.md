# 07. 일정 관리 (Roadmap Management)

## 목적

프로젝트의 주요 마일스톤과 상위 일정을 관리하여 전체적인 진행 상황을 파악합니다. PDFrend는 '성장 중심의 단계별 개발'을 지향하므로, 각 마일스톤은 기능 구현뿐 아니라 **설계 학습, 의사결정 기록, 정책 수립**이 함께 이루어지는 단위로 구성합니다.

```text
                 DB 스키마             주석 CRUD             WebSocket            성능 튜닝
                 파일 업로드/다운로드   OCR 연동              버전 관리            인프라 이관
```

## 일정 운영 원칙

### 스프린트 단위

각 스프린트는 **1~1.5주** 단위로 운영합니다. 스프린트 시작 시 해당 작업의 이슈를 GitHub Issues에 등록하고, 완료 시 관련 관리 문서(ADR, 정책, 학습 기록 등)를 함께 업데이트합니다.

### 단계별 점검 항목

매 스프린트 완료 시 다음 항목을 점검합니다:

| 관리 영역 | 점검 내용 |
| :--- | :--- |
| 정책 관리 | 새로운 정책이 필요한가? 기존 정책에 변경이 있는가? |
| 기획 관리 | PRD에 반영할 변경 사항이 있는가? |
| 설계 관리 | 아키텍처나 DB 스키마에 변경이 있는가? |
| 기술 관리 | 새로운 기술 스택이 추가되었는가? |
| 심화학습 관리 | 이번 스프린트에서 학습한 핵심 내용은 무엇인가? |
| 의사결정 관리 | ADR로 기록할 결정이 있었는가? |
| 이슈 관리 | 미해결 이슈가 남아있는가? 새로운 이슈가 발견되었는가? |

### 일정 조정 기준

일정은 고정이 아닌 **가이드라인**입니다. 다음 상황에서 일정을 조정합니다:

1. **기술적 난관**: 예상보다 복잡한 구현이 필요한 경우, 해당 스프린트를 연장하고 ADR에 사유를 기록합니다.
2. **요구사항 변경**: 기획 변경이 발생하면 영향 범위를 분석하고 로드맵을 재조정합니다.
3. **학습 시간 확보**: '성장 중심' 프로젝트이므로, 새로운 기술을 충분히 이해하는 데 필요한 시간을 확보합니다.
