## ADR-004: 백엔드 기술 스택 및 초기 프로젝트 구조 결정

### 1. 배경
PDFrend 프로젝트의 백엔드 기술 스택으로 Node.js/Python 대신 JVM 기반의 코프링(Kotlin + Spring Boot)이 제안되었으며, 이에 대한 리스크 검토 결과 안정성과 생산성 측면에서 적합하다고 판단되었다. 이에 따라 코프링 기반의 백엔드 프로젝트 초기 구조를 정의하고 개발을 시작한다.

### 2. 결정
코프링(Kotlin + Spring Boot)을 백엔드 주력 기술 스택으로 채택하며, 초기 프로젝트 구조는 다음과 같이 구성한다.

### 3. 상세 결정
*   **기술 스택**: Kotlin, Spring Boot (최신 안정 버전), Gradle (빌드 도구)
*   **주요 의존성**: Spring Web, Spring Data JPA, PostgreSQL Driver, RabbitMQ Client, Spring WebSocket
*   **프로젝트 구조**: `src/main/kotlin` 아래에 `com.pdfrend` 패키지를 기본으로 하며, `config`, `controller`, `service`, `repository`, `domain`, `dto` 등의 계층으로 분리한다.
*   **초기 설정**: `application.yml`을 통해 개발 환경별 설정을 관리하고, `build.gradle.kts`에 필요한 의존성을 명시한다.

### 4. 장점
*   **생산성**: Kotlin의 간결한 문법과 Spring Boot의 강력한 기능으로 빠른 개발이 가능.
*   **안정성**: JVM 기반으로 안정적인 운영 환경 제공.
*   **확장성**: MSA 전환 시 유리한 모듈화 및 확장성.
*   **기술 학습**: jk님의 JVM 기반 기술 학습 목표에 부합.

### 5. 단점
*   초기 설정 및 환경 구성에 다소 시간이 소요될 수 있음.

### 6. 대안 고려
*   **Node.js/Python**: 초기 기획에서 고려되었으나, jk님의 JVM 기반 학습 목표와 장기적인 안정성 및 확장성 측면에서 코프링이 더 적합하다고 판단.

### 7. 상태
Accepted

## ADR-005: 백엔드 모듈화 전략 (초기)

### 1. 배경
향후 MSA(Microservices Architecture)로의 전환 가능성을 염두에 두고, 초기 단계부터 적절한 모듈 분리 기준을 수립하여 효율적인 개발 및 유지보수를 도모한다.

### 2. 결정
초기 모듈화는 **도메인 기반 설계(Domain-Driven Design)** 원칙을 따르며, 핵심 도메인별로 패키지를 분리하고, 인프라 관련 코드는 별도 모듈 또는 패키지로 관리한다.

### 3. 상세 결정
*   **도메인 분리**: `document`, `user`, `annotation`, `sync` 등 핵심 도메인별로 최상위 패키지를 분리한다.
*   **계층형 아키텍처**: 각 도메인 내부는 `controller`, `service`, `repository`, `domain` 등의 계층으로 구성한다.
*   **인프라 분리**: 데이터베이스 접근(`jpa`), 메시징(`rabbitmq`), 웹소켓(`websocket`) 등 인프라 관련 코드는 별도의 `infra` 패키지 또는 모듈로 분리하여 도메인 로직과의 의존성을 최소화한다.
*   **모듈 관리 기준**: 현재는 단일 모놀리식 프로젝트 내에서 패키지 분리를 통해 모듈화를 시작하고, 추후 서비스 규모가 커지면 멀티 모듈 프로젝트 또는 독립적인 마이크로서비스로 분리하는 것을 고려한다.

### 4. 장점
*   **유지보수성**: 도메인별 책임이 명확하여 코드 이해 및 수정 용이.
*   **확장성**: 특정 도메인의 기능 확장 시 다른 도메인에 미치는 영향 최소화.
*   **MSA 전환 용이**: 초기부터 도메인 경계를 명확히 하여 향후 MSA 전환 비용 절감.

### 5. 단점
*   초기 설계 및 패키지 구조 결정에 추가적인 노력이 필요.

### 6. 대안 고려
*   **기능별 분리**: 특정 기능 단위로 모듈을 분리하는 방식. 도메인 간의 강한 결합이 발생할 수 있어 장기적인 확장성에 불리하다고 판단.

### 7. 상태
Accepted

### 8. 관련 이슈
*   #1: 백엔드 프로젝트 초기화 및 기본 설정

## ADR-006: Git 브랜치 전략 (이슈 기반 독립적 배포)

### 1. 배경
프로젝트의 안정적인 개발 흐름을 유지하고, 코드 변경 이력을 명확히 관리하며, 효율적인 협업을 위해 Git 브랜치 전략을 수립한다. 특히, jk님의 피드백을 반영하여 이슈 중심의 독립적 배포 흐름을 따르는 전략을 채택한다.

### 2. 결정
이슈 기반의 독립적 배포 모델을 따르는 Git 브랜치 전략을 채택한다. 주요 브랜치는 `main` (운영), `stg` (스테이징), `dev` (개발)로 구성하며, 모든 기능 개발은 `main`에서 생성된 `feat/#이슈번호` 브랜치에서 진행한다. 개발 완료 후 `dev`, `stg`, `main`으로 각각 머지하며, 충돌 발생 시 `conflict/#이슈번호` 브랜치를 활용하여 해결한다.

### 3. 상세 결정
*   **주요 브랜치**: `main`, `stg`, `dev`
*   **작업 브랜치**: `feat/#이슈번호-기능명` (항상 `main`에서 생성)
*   **머지 흐름**: `feat/#이슈번호` -> `dev` -> `stg` -> `main` (각각 Pull Request를 통해 머지)
*   **충돌 해결**: 머지 충돌 발생 시, 대상 브랜치에서 `conflict/#이슈번호-대상브랜치` 임시 브랜치를 생성하여 충돌 해결 후 다시 대상 브랜치로 머지.
*   **명명 규칙**: 브랜치 및 커밋 메시지는 [PDFrend_Git_Strategy_Guide.md](./PDFrend_Git_Strategy_Guide.md)에 명시된 규칙을 따른다.

### 4. 장점
*   **안정성**: `main` 브랜치의 안정성을 유지하며, 단계별 검증을 통해 운영 환경의 위험 최소화.
*   **이력 관리**: 이슈 기반의 명확한 브랜치 및 커밋 이력으로 변경 사항 추적 용이.
*   **협업 효율성**: 독립적인 작업 브랜치와 명확한 머지 흐름으로 협업 시 혼란 감소.
*   **학습 효과**: Git 워크플로우에 대한 이해 증진 및 실무 적용 능력 향상.

### 5. 단점
*   초기 학습 및 적응 기간 필요.
*   잦은 머지 작업으로 인한 번거로움 발생 가능성.

### 6. 대안 고려
*   **Git Flow**: 복잡한 브랜치 구조로 초기 프로젝트에 과도하다고 판단.
*   **GitHub Flow**: 단순하지만, `stg` 환경의 명확한 분리가 어려움.

### 7. 상태
Accepted

### 8. 관련 이슈

## ADR-007: 초기 백엔드 기술 스택 재검토 (코프링 vs Node.js)

### 1. 배경
PDFrend 프로젝트의 백엔드 기술 스택으로 초기에 코프링(Kotlin + Spring Boot)을 제안했으나, 다음 요소들을 재검토할 필요가 있다:
- 초기 개발 속도 vs 장기 성능 트레이드오프
- 도메인 주도 설계(DDD) 적용 가능성
- 프로젝트 특성(복잡한 버전 관리, 실시간 동기화, JSONB 활용)
- 개발 비용 및 학습곡선

### 2. 대안 비교

#### 코프링(Kotlin + Spring Boot)
**장점:**
- 도메인 모델 표현 명확 (엔티티, 값 객체, 집합 등)
- 성능 우수 (JVM 최적화, 멀티스레딩)
- 엔터프라이즈 검증 (20년 이상 안정성)
- 복잡한 비즈니스 로직 구현 용이
- 트랜잭션 관리 안정적

**단점:**
- 초기 개발 속도 느림 (보일러플레이트 코드 많음)
- 학습곡선 가파름 (JVM, Spring 생태계 이해 필요)
- 메모리 오버헤드 (JVM 기동 시간, 메모리 사용량)
- 빌드 시간 길음

#### Node.js + Express + Drizzle ORM
**장점:**
- 빠른 개발 속도 (간결한 문법, 프로토타이핑 용이)
- 메모리 효율 (낮은 메모리 사용량, 빠른 기동)
- PostgreSQL JSONB 최적화 (완벽한 지원)
- 타입 안전성 (TypeScript + Drizzle의 완벽한 타입 추론)
- 실시간 기능 간단한 구현 (WebSocket 네이티브 지원)
- 배포 용이 (컨테이너화 간단, 클라우드 환경 최적화)

**단점:**
- 성능 한계 (대규모 트래픽에서 JVM 대비 성능 저하 가능)
- 메모리 누수 위험 (비동기 콜백 체인)
- 도메인 모델 표현 덜 명시적
- 관계 관리 수동 처리 필요 (N:M 관계 등)
- 복잡한 트랜잭션 관리 어려움

### 3. DDD 관점 비교

| 항목 | 코프링 | Node.js |
| :---: | :--- | :--- |
| **엔티티** | @Entity 애노테이션 | 스키마 + 타입 정의 |
| **값 객체** | @Embeddable | 중첩 타입 |
| **집합(Aggregate)** | @OneToMany 관계 | 수동 쿼리 조합 |
| **저장소(Repository)** | JpaRepository 상속 | Drizzle 쿼리 빌더 |
| **도메인 이벤트** | Spring Events | 수동 구현 |
| **명시성** | 매우 높음 | 중간 |

### 4. PDFrend 프로젝트 특성 분석

**복잡한 버전 관리:**
- 메이저/마이너 버전, 브랜치/머지 모델
- 코프링: 명확한 도메인 모델로 표현 용이
- Node.js: 타입으로 표현 가능하나 덜 명시적

**실시간 동기화:**
- WebSocket 기반 협업
- 코프링: Spring WebSocket 지원, 복잡한 설정
- Node.js: WebSocket 네이티브 지원, 간단한 구현

**JSONB 활용:**
- 주석, 메타데이터 관리
- 코프링: @Column(columnDefinition = "jsonb") 지원
- Node.js: Drizzle JSONB 타입 완벽 지원

**성능 요구사항:**
- 초기: 사용자 수 적음 (100~1000명)
- 향후: 대규모 협업 환경 (10000명+)

### 5. 최종 결정

**Phase 1-2 (MVP 개발, 3~4개월): Node.js + Express + Drizzle ORM 사용**

**이유:**
1. **빠른 시장 진출**: 2~3주 내 핵심 기능 구현 가능
2. **PostgreSQL JSONB 최적화**: 주석, 메타데이터 관리 효율적
3. **실시간 기능**: WebSocket 네이티브 지원, 간단한 구현
4. **비용 효율**: 메모리 절감, 클라우드 배포 비용 낮음
5. **학습 기회**: jk님이 TypeScript + PostgreSQL 학습 가능

**Phase 3-4 (고도화, 향후 검토): 코프링으로 마이그레이션 검토**

**마이그레이션 트리거:**
- 성능 이슈 발생 (예: 1000+ 동시 사용자)
- 복잡한 비즈니스 로직 강화 필요
- 도메인 모델 고도화 필요
- 엔터프라이즈 안정성 확보 필요

**마이그레이션 전략:**
- 데이터베이스 스키마 유지 (PostgreSQL 동일)
- API 계약 유지 (REST 또는 GraphQL)
- 점진적 마이그레이션 (모듈별 전환)

### 6. 장점
- 빠른 MVP 완성 (3개월 내)
- 초기 개발 비용 절감 (30~40%)
- 유연한 아키텍처 마이그레이션 경로
- jk님의 기술 역량 강화 (TypeScript, PostgreSQL, 나중에 Kotlin)

### 7. 단점
- 향후 대규모 트래픽 대비 마이그레이션 필요 가능성
- 도메인 모델 재설계 필요 (Phase 3-4)
- 초기 설계 단계에서 명확한 도메인 경계 정의 필요

### 8. 상태
Accepted

## ADR-008: Git 브랜치 전략 토큰 효율화 (순차적 승격 모델)

### 1. 배경
초기 Git 브랜치 전략(feat → dev, stg, main 각각 독립 머지)에서 발생하는 토큰 소모량 증가 문제를 해결하기 위해 순차적 승격 모델을 도입한다.

### 2. 문제점
**기존 전략 (독립적 머지):**
```
feat/#001 → dev (1회 머지)
feat/#001 → stg (1회 머지)
feat/#001 → main (1회 머지)
총 3회 머지 = 3배 토큰 소모
```

### 3. 결정
**순차적 승격 모델(Promotion Model) 도입:**
```
feat/#001 → dev (1회 머지)
dev → stg (fast-forward)
stg → main (fast-forward)
총 1회 머지 + 2회 fast-forward = 토큰 50~60% 절감
```

### 4. 상세 결정
**워크플로우:**
1. 작업 이슈 등록 (#001)
2. main에서 feat/#001 브랜치 생성
3. 작업 완료 후 feat/#001 → dev 머지 (Pull Request)
4. dev → stg 승격 (fast-forward, 자동화 가능)
5. stg → main 승격 (fast-forward, 자동화 가능)
6. Hotfix 필요 시: main → dev 역방향 머지

**Hotfix 처리:**
```
hotfix/#002 (main에서 생성)
hotfix/#002 → main (1회 머지)
main → stg (fast-forward)
stg → dev (fast-forward)
```

### 5. 장점
- **토큰 효율화**: 50~60% 토큰 소모 절감
- **이력 명확화**: 각 환경의 배포 이력이 명확함
- **환경별 안정성**: dev → stg → main 순으로 코드 안정성 증가
- **자동화 용이**: fast-forward는 자동화 가능

### 6. 단점
- 브랜치 구조 이해 필요
- dev/stg에서 긴급 수정 시 역방향 머지 필요

### 7. 상태
Accepted

### 8. 관련 이슈
- #2: Git 브랜치 전략 수립 및 저장소 재정리
