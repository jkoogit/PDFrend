## 3. 영역별 리뷰 결과

### 3.1 정책(Policy) 리뷰
- 긍정 평가
  - 인증 주체를 `회원 UUID`로 고정하고, 이메일/간편인증을 보조 인증수단으로 분리한 구조는 계정 주체 혼선을 줄이는 데 효과적이다.
  - 인증수단 고유성(이메일 unique, `(provider, subKey)` unique), 자동 병합 금지, 재인증/알림 제한 정책이 분리되어 운영 규칙 해석이 명확하다.
  - 용어 표준화(로컬 계정 → 회원계정(내부 UUID)) 선언으로 문서 간 용어 충돌 완화 기반이 마련되어 있다.
- 보완 포인트
  - 이메일 역할(로그인 수단 vs 알림 채널) 의미가 하위 문서까지 동일하게 적용되는지 주기적 교차점검이 필요하다.

### 3.2 기획(Planning) 리뷰
- 긍정 평가
  - 기능별 구현설계서에서 `기본 id/pw + 부가 인증수단(간편인증/이메일)` 구조와 `인증수단 우선순위 없음`이 명시되어 사용자 경험 방향이 분명하다.
- 보완 포인트
  - 일부 작업분석 문서에 `비용 제약 기반 이메일 우선` 관점이 강하게 남아 `최종안`과 `단계적 실행안`이 혼재되어 있다.
  - 커뮤니케이션 오류 방지를 위해 기획 문서군을 `최종 목표 아키텍처`와 `현재 릴리즈 단계(비용 제약)`로 명확히 분리해야 한다.

### 3.3 기술검토(Technical) 리뷰
- 긍정 평가
  - 인증수단 원본/링크/이벤트 분리 데이터모델과 유니크 제약 방향은 정책과 정합성이 높다.
  - 제공자별 API 체크리스트 및 `이메일은 로그인 필수조건 아님` 명시로 구현 시 해석 편차를 줄이는 장치가 있다.
- 보완 포인트
  - 저비용 전략 문서의 `간편인증 제외 + 이메일 중심` 서술이 최종안 문서와 충돌할 여지가 있다.
  - 문서 상단에 `목표 상태(간편인증 포함)` vs `현 릴리즈 범위(이메일 우선)` 경계 주석을 통일해야 한다.

### 3.4 이슈관리(Issue) 리뷰
- 긍정 평가
  - 이슈 템플릿에서 정책 링크/식별 기준/이메일 역할/참조문서를 연결해 실행 문서로서 실용성이 높다.
  - `이메일 미인증 상태 간편로그인`, `인증수단 삭제 시 최소 1개 유지` 등 운영 이슈를 질문 항목으로 끌어온 점이 유의미하다.
- 보완 포인트
  - 다음 단계에서는 체크리스트를 실제 개발 태스크(백엔드/프론트/운영)로 분해하고 담당자/완료조건(DoD)을 부여해야 한다.

### 3.5 의사결정(Decision) 리뷰
- 확정에 가까운 항목
  1) 계정 주체 UUID
  2) 인증수단 고유키
  3) 자동 병합 금지
  4) 로그인 시 인증수단 우선순위 없음(사용자 선택)
- 추가 결정이 필요한 항목
  1) 재인증 시 교체 트랜잭션 상세
  2) refresh token 저장 위치
  3) 제공자별 이메일/식별정보 편차 대응
  4) 용어 전사 반영 범위
- 실행 제안
  - 다음 채팅에서 위 4개 미확정 항목을 ADR 단문 세트로 확정해 구현 속도를 높인다.

## 4. 종합 평가
- 강점
  - 정책/데이터모델/실행 템플릿의 기본 뼈대는 충분히 안정적이다.
- 핵심 리스크
  - `최종안(간편인증 포함)`과 `비용단계(이메일 우선)` 문장 혼재로 문서별 해석 차이가 발생할 수 있다.
- 우선순위 후속 액션
  1. 문서 상단에 `문서 성격 라벨(목표안/단계안)` 통일
  2. 오픈 의사결정 4건을 ADR로 확정
  3. 이슈 템플릿 체크리스트를 구현 태스크 + DoD로 분해
