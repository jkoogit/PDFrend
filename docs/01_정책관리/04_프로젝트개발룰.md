# 04. 프로젝트 개발 룰 (Development Rules)

## 개요
매 작업 단계에서 코드 품질과 관리 체계를 유지하기 위해 반드시 준수해야 할 필수 요건입니다.

## 1. Git & PR 워크플로우
- [ ] 모든 작업은 GitHub Issue 생성 후 시작한다.
- [ ] 이슈 타이틀에는 반드시 이슈 번호를 포함한다 (예: `[이슈 #2] DB 스키마 설계`).
- [ ] 작업 브랜치는 `main`에서 생성하며 `feature/#이슈번호-작업명` 형식을 사용한다.
- [ ] 작업 완료 후 `dev` 브랜치로 **Pull Request(PR)**를 생성한다.
- [ ] PR 내에서 Manus의 **셀프 코멘트**와 사용자의 리뷰를 거친 후 머지한다.
- [ ] 다음 작업 시작 전, 이전 작업 소스를 `stg` 및 `main`으로 승격시킨다.

## 2. 문서화 및 학습 (Hybrid Review)
- [ ] 새로운 기술 결정이나 정책 변경 시 관련 문서를 즉시 업데이트한다.
- [ ] 이슈 발생 시 `{YYMMDD}_{순번}_{문제}_{원인}.md` 형식으로 리포트를 작성한다.
- [ ] 작업 완료 후 **Deep Dive 보고서**와 **심화 학습 가이드**를 작성한다.
- [ ] 리뷰 보고서 파일명 규칙(`날짜_순번_리뷰구분_#이슈번호_이슈제목`)을 준수한다.

## 3. 코드 품질 및 검증
- [ ] 각 폴더의 진입점 파일은 `index_{역할}.ts` 형식을 사용한다. (예: `index_svr.ts`, `index_db.ts`, `index_sch.ts`) 이는 파일 목록에서 식별력을 높이고 구성의 시작점임을 명시하기 위함이다.

- [ ] `npx tsc --noEmit`을 통해 타입 오류가 없음을 확인한다.
- [ ] ESLint를 실행하여 린트 에러가 없음을 확인한다.
- [ ] 새로운 API 추가 시 tRPC 헬스체크 또는 단위 테스트를 통해 동작을 검증한다.
- [ ] Cloudtype 배포 후 실제 URL에서 정상 동작 여부를 최종 확인한다.

## 4. 환경 및 보안
- [ ] 민감 정보(비밀번호, API 키)는 절대 코드나 문서에 직접 기록하지 않는다.
- [ ] 환경 변수는 `docs/01_정책관리/03_환경변수관리대장.md`에 항목만 등록하여 관리한다.
- [ ] 모든 브랜치(`dev`, `stg`, `main`)의 동기화 상태를 주기적으로 점검한다.
