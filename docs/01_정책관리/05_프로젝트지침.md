# 05. 프로젝트 지침 (Project Instructions)

## 개요
본 문서는 PDFrend 프로젝트의 일관성과 품질을 유지하기 위해 AI 에이전트(Manus)가 반드시 준수해야 할 핵심 지침을 정의합니다. 모든 작업은 본 지침을 최우선으로 따릅니다.

## 1. 문서 및 이슈 관리 규칙
- **이슈 기반 개발**: 모든 작업은 GitHub Issue 생성 후 시작하며, 타이틀에 반드시 이슈 번호를 포함한다. (예: `[이슈 #N] 제목`)
- **이슈 리포트**: 문제 발생 시 `{YYMMDD}_{순번}_{문제}_{원인}.md` 형식으로 `docs/08_이슈관리/`에 기록한다.
- **리뷰 관리**: 리뷰 보고서는 `{YYMMDD}_{순번}_{리뷰구분}_#이슈번호_이슈제목(10자).md` 형식으로 `docs/09_리뷰관리/YYYYMM/DD/` 폴더에 저장한다.
- **파일명**: 모든 문서는 한글 파일명을 사용하며, 번호 체계(00_, 01_ 등)를 준수한다.

## 2. Git 및 브랜치 전략
- **브랜치 구조**: `dev` (개발), `stg` (검증), `main` (운영) 체계를 유지한다.
- **워크플로우**: `feature` 브랜치 작업 -> `dev` 머지 및 검증 -> `stg`/`main` 승격(Promotion) 순으로 진행한다.
- **계정**: `jkman` 계정을 사용하여 활동하며, 커밋 메시지에 이슈 번호를 참조한다.

## 3. 기술 및 설계 원칙
- **아키텍처**: DDD(Domain-Driven Design) 폴더 구조를 준수한다 (`api`, `domain`, `infrastructure`, `shared`).
- **기술 스택**: Node.js + Express + tRPC + TypeScript + Drizzle ORM 환경을 유지한다.
- **설정 관리**: 환경별 설정(`common`, `dev`, `stg`, `prd`)을 분리하고, 민감 정보는 환경 변수로 관리한다.

## 4. 품질 및 보안 지침
- **정적 분석**: 매 작업 완료 전 `tsc --noEmit` 및 `ESLint` 검사를 수행한다.
- **보안**: 민감 정보는 절대 코드나 문서에 직접 기록하지 않으며, `03_환경변수관리대장.md`에 마스킹 처리하여 항목만 관리한다.
- **현행화**: 대화 중 결정된 사항이나 기술적 변경점은 즉시 관련 문서에 반영한다.

## 5. 작업 재개 절차
- 샌드박스 재시작 시 Cloudtype 서비스 상태를 먼저 확인하고, 필요 시 서비스를 재개한 후 작업을 시작한다.
