# [시스템 정책서] PDFrend: 운영 및 기능 정책

본 문서는 PDFrend 서비스의 핵심 운영 및 기능 정책을 정의합니다. 각 정책에는 고유한 식별자(Policy ID)가 부여되며, 관련 기획 및 설계 문서에서 참조됩니다.

## 1. 스토리지 및 용량 관리 정책

### Policy-STG-001: 기본 사용자 할당 용량
*   **정책**: 신규 가입 사용자에게는 기본적으로 1GB의 문서 업로드 용량이 할당된다.
*   **관리**: 시스템 관리 화면에서 기본 할당 용량을 설정 및 변경할 수 있다.

### Policy-STG-002: 사용자별 용량 설정 및 제한
*   **정책**: 시스템 관리자는 사용자 관리 화면에서 개별 사용자에게 문서 업로드 용량을 할당할 수 있다.
*   **제한**: 사용자의 현재 사용 용량이 할당된 용량을 초과하거나, 할당 용량이 현재 사용 용량보다 낮게 변경될 경우, 추가 파일 업로드를 제한한다.
    *   **안내**: 용량 제한 시 사용자에게 적절한 안내 메시지(예: 
현재 사용 용량이 할당 용량을 초과하여 추가 업로드가 제한됩니다. 용량을 확보하거나 관리자에게 문의하세요.)를 표시한다.

### Policy-STG-003: 공유 문서 용량 계산 정책
*   **정책**: 다른 사용자로부터 공유받은 문서는 해당 사용자의 문서 업로드 용량 계산에 포함되지 않는다.
*   **적용**: 사용자의 총 사용 용량은 본인이 직접 업로드하거나 생성한 문서의 용량만을 기준으로 한다.

## 2. 버전 및 동시성 관리 정책

### Policy-VER-001: 버전 관리 모델
*   **정책**: Git 방식의 브랜치/머지 모델을 도입하여 문서 버전을 관리한다. 기본적으로 'Main' 버전을 유지하며, 동시 작업 시 각 사용자의 작업을 'Branch(Draft)'로 관리하고, 최종적으로 소유자가 'Merge' 여부를 결정한다.

### Policy-VER-002: 자동 저장 및 체크포인트
*   **정책**: 모든 주석 수정은 실시간으로 서버에 반영되며 '작업 이력'으로 관리된다. 사용자가 명시적으로 '버전 생성'을 클릭할 때만 버전 번호가 부여된다.

### Policy-VER-003: 동시 작업자 처리 (버전 생성 시)
*   **정책**: 한 작업자가 '버전 생성'을 진행할 경우, 다른 동시 작업자들에게는 새로운 버전으로 자동 업데이트되어 작업이 계속되도록 안내 메시지를 표시한다.

## 3. 오프라인 동기화 및 충돌 해결 정책

### Policy-OFF-001: 오프라인 동기화 충돌 해결
*   **정책**: 오프라인 작업 후 온라인 전환 시, 'Last Write Wins'를 기본으로 하되, 동일 영역에 대한 충돌 발생 시 사용자에게 두 버전을 시각적으로 비교하여 선택하게 하는 'Conflict UI'를 제공한다.

### Policy-OFF-002: 문서 메타정보 활용
*   **정책**: 파일명뿐만 아니라 문서 내 메타정보(Document_ID, Current_Version, Last_Modified_By, Last_Timestamp)를 활용하여 문서의 동일성, 최신성, 수정 여부를 정확히 판별한다.

## 4. DB 구조 및 데이터 관리 정책

### Policy-DB-001: 주석 및 목차 데이터 저장
*   **정책**: 주석과 목차 데이터는 PostgreSQL의 JSONB 타입을 활용하여 저장한다. 이를 통해 특정 작성자의 주석 검색이나 특정 페이지 범위의 주석 추출 속도를 최적화한다.

## 5. 인증 식별/계정 매핑 정책

### Policy-AUTH-001: 계정 식별 기준
*   **정책**: 회원의 내부 식별자는 `UUID`를 사용한다. 인증수단(이메일/간편인증)은 계정 식별 보조수단이며, 최종 계정 주체는 UUID다.

### Policy-AUTH-002: 인증수단별 고유키
*   **정책**: 이메일 인증은 `email` 유니크로 관리한다. 간편인증은 provider별 고유 `subKey`를 사용하여 `(provider, subKey)` 복합 유니크로 관리한다.

### Policy-AUTH-003: 이메일 인증과 간편인증의 병합 원칙
*   **정책**: 이메일 인증과 간편인증은 검증 정보의 성격이 다르므로 자동 동일인 병합을 수행하지 않는다.
*   **적용**: 인증수단 추가 연결은 명시적 사용자 검증 절차를 통과한 경우에만 허용한다.

### Policy-AUTH-004: 용어 표준화
*   **정책**: 기존 문서의 `로컬 계정` 용어는 `회원계정(내부 UUID 계정)`으로 통일한다.
*   **목적**: 가입 계정과 인증수단(이메일/간편인증)의 역할 혼선을 제거하고 설계/개발/운영 문서 정합성을 유지한다.



### Policy-AUTH-005: 복수 인증수단 관리
*   **정책**: 하나의 회원계정(UUID)은 이메일 인증, 간편인증(전화 기반 포함) 등 복수 인증수단을 연결하여 관리할 수 있다.

### Policy-AUTH-006: 가입 충돌 시 로그인 유도
*   **정책**: 이미 등록된 이메일/전화번호/간편인증키로 가입 시도 시 신규 계정을 생성하지 않는다.
*   **적용**: `등록된 회원입니다. 로그인하시겠습니까.` 메시지로 로그인 전환을 유도한다.

### Policy-AUTH-007: 인증갱신 및 알림 발송 제한
*   **정책**: 로그인 후 사용자정보에서 이메일/전화번호 재인증을 허용한다. 재인증 갱신은 중복체크 트랜잭션으로 처리한다.
*   **적용**: 알림 발송 대상은 인증 완료된 연락처 중 `is_primary=true`(최종 인증 연락처) 1건으로 제한한다.
*   **예외처리**: 번호 변경 이력으로 과거 번호가 유효 상태로 남아있더라도, 중복 발송 방지를 위해 비-Primary 연락처에는 알림을 발송하지 않는다.

## 6. 데이터 사전 및 코드 현행화 정책

### Policy-DATA-001: 데이터 사전 표준 관리
*   **정책**: 단어사전, 용어사전, 도메인사전, 공통코드사전을 표준 산출물로 관리한다.
*   **적용**: 컬럼 명명/타입/코드값 변경 시 사전 문서를 먼저 또는 동시에 수정한다.

### Policy-DATA-002: 테이블-문서 동시 현행화
*   **정책**: 추후 단어/용어/도메인/공통코드 정보는 DB 테이블로도 관리한다.
*   **적용**: DB 테이블과 문서 사전은 동일 버전으로 함께 현행화한다(문서 단독 변경 금지).

### Policy-DATA-003: 코드값 중앙통제
*   **정책**: 상태/유형/결과 코드는 공통코드사전의 그룹코드/상세코드를 기준으로 사용한다.
*   **적용**: 신규 코드 추가 시 공통코드사전 등록 후 테이블 반영한다.
