# [기술 검토] PDFrend 백엔드 및 인프라 구성안

## 1. 백엔드 기술 스택 검토: 코프링 (Kotlin + Spring Boot)

기존 Node.js/Python 대신 JVM 기반의 코프링(Kotlin + Spring Boot)으로 백엔드를 구현하는 방안을 검토합니다. 코프링은 Spring Boot의 강력한 생태계와 Kotlin의 현대적인 언어 특성을 결합하여 생산성과 안정성을 높일 수 있는 좋은 선택입니다.

### 1.1 코프링의 장점
*   **생산성**: Kotlin의 간결한 문법, Null Safety, 코루틴(Coroutines)을 통한 비동기 프로그래밍 지원 등으로 개발 생산성이 향상됩니다.
*   **안정성**: JVM 기반으로 안정적인 운영 환경을 제공하며, Spring Boot의 풍부한 기능과 검증된 라이브러리를 활용할 수 있습니다.
*   **성능**: Java와 유사한 수준의 성능을 제공하며, 코루틴을 통해 고성능 비동기 처리가 가능합니다.
*   **Spring 생태계 활용**: Spring Data JPA, Spring Security, Spring WebFlux 등 기존 Spring Boot의 모든 기능을 Kotlin으로 동일하게 사용할 수 있습니다.

### 1.2 코프링의 리스크 및 고려사항
*   **학습 곡선**: 기존 Java 개발자에게는 Kotlin 문법과 코루틴에 대한 학습이 필요할 수 있습니다.
*   **초기 설정 복잡성**: 프로젝트 초기 설정 시 Java 대비 Kotlin 관련 의존성 관리 및 빌드 설정에 약간의 추가 작업이 필요할 수 있습니다.
*   **버전 호환성**: Spring Boot와 Kotlin 버전 간의 호환성을 주의해야 합니다. 일반적으로 Spring Boot는 특정 Kotlin 버전을 권장하며, 최신 버전 사용 시 호환성 문제를 미리 확인해야 합니다 [1].
    *   **권장**: Spring Boot 공식 문서에서 권장하는 Kotlin 버전을 따르는 것이 가장 안전합니다. 예를 들어, Spring Boot 3.x 버전은 Kotlin 1.x 또는 2.x 버전을 지원하며, Spring Boot 4.x (예정)에서는 Kotlin 2.x에 대한 지원이 강화될 것으로 예상됩니다 [2].
*   **RabbitMQ 및 WebSocket 연동**: Spring Boot는 RabbitMQ (Spring AMQP) 및 WebSocket (Spring WebSocket)에 대한 강력한 지원을 제공하므로, Kotlin으로 구현 시에도 큰 리스크 없이 연동 가능합니다. 코루틴을 활용하여 비동기 메시지 처리 및 실시간 통신을 효율적으로 구현할 수 있습니다.

### 1.3 결론 및 권장 사항
코프링은 PDFrend 프로젝트의 요구사항(비동기 저장, 실시간 동기화)을 충족시키면서 높은 생산성과 안정성을 제공할 수 있는 좋은 선택입니다. 학습 곡선과 초기 설정의 복잡성은 존재하지만, 장기적인 관점에서 이점이 더 크다고 판단됩니다. **Spring Boot의 최신 안정 버전과 해당 버전에서 공식적으로 지원하는 Kotlin 버전을 사용하는 것을 권장합니다.**

## 2. 저비용 인프라 구성안 (클라우드 대체)

클라우드 서비스(S3) 대신 NAS나 Ubuntu 설치 노트북을 활용한 저비용 인프라 구성안을 제안합니다. 이는 초기 비용을 절감하고 데이터 주권을 확보하는 데 유리하지만, 확장성, 안정성, 유지보수 측면에서 고려할 점이 있습니다.

### 2.1 스토리지 (S3 대체)
*   **NAS (Network Attached Storage)**: 
    *   **장점**: 대용량 저장 공간, 비교적 쉬운 설정, 데이터 백업 및 공유 기능 내장.
    *   **활용**: PDF 원본 파일, 페이지별 이미지(JPG) 저장에 적합. SMB/NFS 프로토콜을 통해 백엔드 서버에서 마운트하여 파일 시스템처럼 사용.
    *   **고려사항**: NAS 자체의 안정성(RAID 구성 여부), 네트워크 대역폭, 외부 접근 시 보안 설정(VPN 또는 포트 포워딩)이 중요.
*   **Ubuntu 서버 (로컬 디스크)**:
    *   **장점**: 추가 하드웨어 비용 없음 (기존 노트북 활용), 직접적인 파일 시스템 제어.
    *   **활용**: NAS와 유사하게 PDF 관련 파일을 저장. Docker Volume 등을 활용하여 백엔드 애플리케이션과 연동.
    *   **고려사항**: 하드웨어 고장 시 데이터 손실 위험(정기적인 백업 필수), 디스크 I/O 성능, 전력 소모.

### 2.2 데이터베이스 (PostgreSQL)
*   **NAS 또는 Ubuntu 서버에 직접 설치**: PostgreSQL을 NAS 또는 Ubuntu 서버에 직접 설치하여 운영합니다.
    *   **장점**: 클라우드 DB 서비스 비용 절감, 데이터베이스에 대한 완전한 제어권.
    *   **활용**: JSONB 타입을 활용하여 OCR 및 주석 데이터를 효율적으로 저장.
    *   **고려사항**: DB 백업 및 복구 전략 수립, 성능 튜닝, 보안 설정(접근 제어, 암호화) 등 DBA 역할이 필요.

### 2.3 메시지 큐 (RabbitMQ)
*   **NAS 또는 Ubuntu 서버에 직접 설치**: RabbitMQ 또한 NAS 또는 Ubuntu 서버에 직접 설치하여 운영합니다.
    *   **장점**: 클라우드 메시지 큐 서비스 비용 절감.
    *   **활용**: 비동기 주석 저장 및 동기화 처리에 활용.
    *   **고려사항**: RabbitMQ 클러스터링 구성(고가용성 필요 시), 모니터링 및 유지보수.

### 2.4 인프라 구성 시 고려사항
*   **가용성 및 안정성**: 단일 서버 구성 시 하드웨어 고장, 네트워크 장애 등으로 서비스 중단 위험이 높습니다. 이중화(HA) 구성은 비용이 증가합니다.
*   **확장성**: 사용자 증가 시 성능 병목 현상이 발생할 수 있으며, 하드웨어 업그레이드 외에는 확장하기 어렵습니다.
*   **보안**: 외부 접근 시 방화벽, VPN, SSL/TLS 암호화 등 철저한 보안 설정이 필수적입니다.
*   **유지보수**: OS, DB, 미들웨어(RabbitMQ) 등에 대한 정기적인 업데이트, 백업, 모니터링 등 운영 부담이 큽니다.

### 2.5 결론 및 권장 사항
저비용 인프라 구성은 초기 비용 절감에 효과적이지만, 서비스의 안정성, 확장성, 보안, 유지보수 측면에서 많은 노력이 필요합니다. 개인 프로젝트나 소규모 서비스에는 적합할 수 있으나, 향후 사용자 증가를 고려한다면 클라우드 서비스로의 전환을 염두에 두어야 합니다. **초기에는 NAS 또는 Ubuntu 서버에 모든 컴포넌트(PostgreSQL, RabbitMQ, 백엔드 애플리케이션)를 Docker 컨테이너로 구성하여 관리의 용이성을 확보하는 것을 권장합니다.**
