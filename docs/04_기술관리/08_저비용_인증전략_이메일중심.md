# 08. 저비용 인증전략 (이메일 중심 전환)

## 1. 배경
이슈 #5 진행 중 비용 제약으로 간편인증/본인인증(유료)을 제외하고, 이메일 중심 인증으로 전략을 전환한다.

## 2. 결정 사항
- 이메일 인증은 간편인증 로그인 필수조건이 아니다. (이메일은 협업/서비스 소식 공유 채널)
1. **간편인증 제외**: 카카오/네이버/구글 소셜로그인 및 본인인증 연동은 현 단계에서 보류
2. **본인인증 제외**: PASS/문자 본인확인 등 유료 본인확인 서비스 보류
3. **이메일 인증 채택**: 회원가입/로그인/재인증은 이메일 기반으로 우선 구현

## 3. 사용자 질문 답변 요약
- 이메일 인증 가능 여부: **가능**
- 핸드폰 번호 검증 우회수단:
  - 완전한 본인확인 대체는 어려움
  - 저비용 대안은 "번호 수집 + 검증 보류" 또는 "운영자 수동 검증" 수준
  - 전화번호를 신뢰 데이터로 쓰려면 결국 SMS/본인확인 비용이 필요

## 4. 현실적인 저비용 대안

### 대안 A. 이메일만 사용 (권장)
- 가입: 이메일 OTP/매직링크
- 로그인: 이메일 OTP/패스워드+이메일 2차 인증
- 장점: 즉시 구현 가능, 비용 최소
- 단점: 전화번호 실명/실소유 검증 불가

### 대안 B. 전화번호는 수집만 하고 미검증 상태 유지
- 프로필에 번호 입력 허용
- `phone_verified=false` 기본값 유지
- 알림은 이메일 우선 발송
- 장점: UX 단순, 비용 없음
- 단점: 번호 기반 보안정책 적용 불가

### 대안 C. 운영자 수동 검증
- 고객센터 확인 후 수동 `phone_verified=true`
- 장점: 소량 사용자에서는 비용 절감
- 단점: 운영부담 큼, 확장성 낮음

## 5. 권장 구현 정책
1. 계정 신뢰도 기준은 `email_verified=true`로 통일
2. 전화번호는 미검증 데이터로 분리 관리
3. 중요 기능(비밀번호 재설정, 계정복구)은 이메일 검증 기반으로만 허용
4. 추후 예산 확보 시 SMS 검증 모듈 추가

## 6. 단계별 실행안
1. 즉시: 이메일 OTP/매직링크 플로우 구현
2. 다음: 계정복구/알림을 이메일 채널로 통합
3. 추후: SMS 검증 비용 견적 후 단계적 도입


## 8. 번외 아이디어 검토: "서브폰 앱으로 SMS 발송 대행"

### 아이디어 요약
- 서브폰(실물 단말)에 앱을 설치하고,
- 앱이 DB에서 발송 요청을 조회해,
- 단말 기본 SMS 기능으로 인증문자를 발송하는 방식.

### 기술적으로 가능한가?
- **가능은 하다.** (안드로이드 앱에서 SMS 발송 권한 사용)
- 다만 iOS는 제약이 크고, 안드로이드도 스토어 정책/권한 심사가 까다롭다.

### 이 방식의 장점
- 초기 소량 트래픽에서는 SMS API 계약 전 임시 운용 가능
- 발신번호가 실제 단말번호로 고정되어 사용자 혼선이 적을 수 있음

### 핵심 리스크 (중요)
1. **신뢰성 리스크**: 단말 배터리/네트워크/앱 강제종료 시 발송 누락
2. **확장성 리스크**: 발송량 증가 시 병목(단말 1~N대 수작업 운영)
3. **보안 리스크**: DB 직접조회 구조는 단말 탈취/토큰 유출 시 대량 오발송 위험
4. **운영 리스크**: 발송 성공/실패 추적, 재시도, 중복발송 제어가 어려움
5. **규제/약관 리스크**: 국가별 스팸/광고/본인확인 규정 위반 가능성 검토 필요

### 결론
- **PoC/내부 테스트 용도**로는 검토 가능
- **상용 본인인증 대체수단으로는 비권장**
- 상용 단계에서는 SMS 게이트웨이(또는 본인확인 서비스)로 전환하는 것이 안전

### 최소 안전장치(임시 운용 시 필수)
1. 단말 앱은 DB 직접 조회 금지 → 서버의 서명된 발송 큐 API만 호출
2. 1회용 토큰 + 짧은 TTL + IP/디바이스 바인딩
3. 발송 요청 idempotency 키 적용(중복발송 방지)
4. 발송 로그/감사로그/재시도 횟수 제한
5. 인증코드 유효시간(예: 3분)과 시도횟수 제한

## 7. 이슈 #5 문서 반영 가이드

## 9. 추가 검토: FCM / 카카오 알림톡 / NCP SMS를 인증 목적으로 쓸 수 있나?

### 9.1 Firebase Cloud Messaging (FCM)
- 성격: **푸시 알림 서비스** (SMS 아님)
- 인증 목적 적합성:
  - 앱 설치 + 푸시 수신이 가능한 사용자에게만 전달 가능
  - 전화번호 소유 검증 수단으로는 부적합(번호 자체를 검증하지 않음)
- 전화번호 수집 가능성:
  - FCM 자체로 전화번호를 수집/검증하지 않음
- 결론:
  - 인증코드 전달 채널(보조 채널)로는 가능
  - **휴대폰 번호 본인확인 대체는 불가**

### 9.2 카카오 알림톡 API
- 성격: 비즈메시지(알림톡), 템플릿 사전 승인 필요
- 인증 목적 적합성:
  - 채널 기반 안내/인증코드 통지는 가능
  - 다만 SMS와 다르며 카카오톡 사용 환경 의존
- 전화번호 수집 가능성:
  - 알림톡 발송 자체는 수신번호를 알아야 가능(입력/보유 필요)
  - 알림톡 API가 번호를 "수집/증명"해주지는 않음
- 결론:
  - 인증 메시지 전달 수단으로는 가능
  - **번호 소유 검증 자체를 대체하지는 못함**

### 9.3 NCP SENS SMS
- 성격: SMS/알림톡 발송 API
- 인증 목적 적합성:
  - OTP 문자 인증 구현 가능(일반적인 방식)
- 전화번호 수집 가능성:
  - SMS OTP로 번호 소유 검증 가능(코드 수신·입력 기반)
- 비용:
  - 신규 포인트는 보통 테스트 목적이며 상용은 유료 과금 전환
- 결론:
  - 이 3개 중 "번호 검증"에 가장 직접적
  - 다만 무료 영구 운영은 어렵고 비용/정책 검토 필요
