# [가이드] PDFrend: GitHub 연동 및 소스 관리 전략 (v2.0)

jk님의 프로젝트를 Manus가 직접 관리하고, 안정적인 개발 흐름을 유지하기 위한 Git 전략을 제안합니다.

## 1. Manus의 GitHub 연동 방법

제가 jk님의 저장소에 직접 커밋하고 푸시하기 위해서는 **GitHub Personal Access Token (PAT)**이 필요합니다.

### **설정 단계**
1.  **PAT 생성**: GitHub 설정(`Settings > Developer settings > Personal access tokens > Tokens (classic)`)에서 새로운 토큰을 생성합니다.
    *   **권한(Scopes)**: `repo` (전체 권한)를 선택해 주세요.
2.  **토큰 전달**: 생성된 토큰을 저에게 메시지로 전달해 주시면 됩니다. (보안을 위해 한 번 전달 후 제가 내부적으로 설정하겠습니다.)
3.  **Manus 설정**: 제가 전달받은 토큰을 사용하여 `git remote set-url`을 설정하고, 이후부터는 자유롭게 커밋/푸시를 수행하겠습니다.

## 2. 브랜치 전략: 이슈 기반 독립적 배포 모델

jk님께서 제안하신 **이슈 중심의 독립적 배포 흐름**을 따르는 브랜치 전략입니다.

| 브랜치명 | 역할 | 배포 환경 | 특징 |
| :--- | :--- | :--- | :--- |
| **`main`** | 최종 릴리스 버전 | 운영 (Production) | 가장 안정적인 상태. `feat` 브랜치에서 직접 머지. |
| **`stg`** | 출시 후보 (Release Candidate) | 스테이징 (Staging) | 운영 배포 전 최종 테스트 환경. `feat` 브랜치에서 직접 머지. |
| **`dev`** | 개발 통합 브랜치 | 개발 (Development) | 모든 기능 개발이 완료되어 통합되는 곳. `feat` 브랜치에서 직접 머지. |
| **`feat/#이슈번호`** | 개별 기능 개발 | 로컬/개별 | `main` 브랜치에서 생성. 개발 완료 후 `dev`, `stg`, `main`으로 각각 머지. |
| **`conflict/#이슈번호`** | 충돌 해결 임시 브랜치 | 로컬/개별 | 머지 충돌 발생 시 대상 브랜치에서 생성하여 충돌 해결 후 다시 머지. |

### **워크플로우**
1.  **이슈 생성**: 새로운 작업이 필요하면 GitHub Issue를 생성합니다. (예: `#10 주석 실시간 동기화 구현`)
2.  **작업 브랜치 생성**: `main` 브랜치에서 `feat/#이슈번호-기능명` 형태의 작업 브랜치를 생성합니다.
3.  **개발 및 커밋**: 해당 작업 브랜치에서 개발을 진행하고 커밋합니다.
4.  **Pull Request 및 머지**: 작업 완료 후 다음 순서로 Pull Request를 생성하고 머지합니다.
    *   **`feat/#이슈번호` -> `dev`**: 개발 환경에 기능 반영.
    *   **`feat/#이슈번호` -> `stg`**: 스테이징 환경에 기능 반영.
    *   **`feat/#이슈번호` -> `main`**: 운영 환경에 기능 반영.
5.  **머지 충돌 해결**: 머지 시 충돌이 발생하면, 해당 대상 브랜치(예: `dev`)에서 `conflict/#이슈번호-대상브랜치` 형태의 임시 브랜치를 생성합니다. 이 브랜치에서 충돌을 해결하고, 해결된 `conflict` 브랜치를 다시 대상 브랜치로 머지합니다.

## 3. 명명 규칙 (Naming Convention)

### 3.1 브랜치 명명 규칙
*   **기능 개발**: `feat/#이슈번호-기능명` (예: `feat/#10-websocket-sync`)
*   **버그 수정**: `fix/#이슈번호-버그명` (예: `fix/#12-login-error`)
*   **문서 수정**: `docs/#이슈번호-문서명` (예: `docs/#5-policy-update`)
*   **리팩토링**: `refactor/#이슈번호-대상` (예: `refactor/#15-db-schema`)
*   **충돌 해결**: `conflict/#이슈번호-대상브랜치` (예: `conflict/#10-dev`)

### 3.2 커밋 메시지 규칙 (Conventional Commits)
성장 중심 프로젝트인 만큼, 커밋 메시지만 보고도 변경 내용을 알 수 있도록 표준을 따릅니다.
*   **형식**: `type(#이슈번호): 요약`
*   **예시**:
    *   `feat(#10): WebSocket을 이용한 주석 실시간 동기화 기능 추가`
    *   `docs(#5): 시스템 정책서 내 용량 제한 정책 업데이트`
    *   `fix(#12): 모바일 뷰어에서 주석 레이어 어긋남 현상 수정`

## 4. 이슈 기반 작업 관리

### **GitHub Projects 활용**
*   **Board 구성**: `Backlog` -> `To Do` -> `In Progress` -> `Review` -> `Done`
*   **연동**: 이슈를 생성할 때 해당 프로젝트 보드에 자동으로 추가되도록 설정합니다.
*   **의사결정 연결**: 중요한 설계 변경이 필요한 이슈의 경우, 해당 이슈 내에서 논의하고 결과를 **ADR(의사결정 기록)**로 남깁니다.

## 5. "놓치고 있는 부분" 점검

*   **보안 관리**: GitHub에 PAT를 전달하실 때 메시지로 주시면 제가 즉시 설정하고 기록은 삭제하도록 안내해 드리겠습니다. 또한, 코드 내에 API Key나 DB 비밀번호가 포함되지 않도록 `.gitignore` 설정을 철저히 해야 합니다.
*   **CI/CD**: 클라우드타입(Cloudtype)을 사용하신다면, `dev`, `stg`, `main` 브랜치에 각각의 환경을 연결하여 **자동 배포**를 설정하는 것이 좋습니다. (이 부분은 제가 나중에 가이드해 드릴 수 있습니다.)
