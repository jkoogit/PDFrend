## 4. 1.3 작업대상 분석 (사전 검토)

### 4.1 현재 구현 상태
- 로컬 인증 API(`signup/login/me/logout`)는 1차 구현됨.
- 토큰 검증, 비밀번호 해시, 보호 프로시저 도입됨.
- 로그아웃 시 토큰 revoke(서버 메모리 denylist) 처리 반영됨.

### 4.2 보완 필요 항목 (사전 점검 결과 + 즉시 보완 가능 범위)
1. **보안 보강**
   - [완료] 로그아웃 revoke 적용
   - [진행 필요] 토큰 회전/재발급 정책
2. **권한 모델 정교화**
   - [잔여업무] 사용자 Role/권한 범위와 CRUD 보호 연계 미완료
   - 선행조건: 역할 모델(예: admin/editor/user) 및 리소스 정책 합의 필요
3. **DB 마이그레이션 정합성**
   - [진행 필요] schema 변경(`password_hash/provider/provider_id`) 마이그레이션 적용 검증
4. **테스트 부족**
   - [진행 필요] 실패 케이스(잘못된 토큰/만료/중복가입/잘못된 비밀번호) 자동화

## 5. 1.4 실행계획 (상세)

### 단계 A: 안정화 (P0)
- A1. 인증 API 예외 처리 정리(오류코드 표준화)
- A2. 문서화된 보안 기본선 확정(토큰 만료, 비밀번호 정책)
- A3. DB 마이그레이션/스키마 동기화 확인

### 단계 B: 보호 연계 (P1)
- B1. 기존 User/Document API에 인증 컨텍스트 연계 전략 정의
- B2. 최소 권한 정책(본인 소유 리소스만 접근) 재점검
- B3. Role 기반 정책은 후속 개발 스프린트에서 잔여업무로 처리

### 단계 C: 검증 확장 (P1)
- C1. 단위 테스트: 해시/토큰 유틸
- C2. 통합 테스트: signup→login→me→logout 흐름
- C3. 인수 테스트: 인증 실패/권한거부 시나리오

### 단계 D: 세션 인계 (P2)

## 7. 의사결정 확정
1. **이슈 번호**: `#4=인증 구현`으로 확정 (세션 고정)
2. **토큰 방식 비교 결론 (HMAC 커스텀 vs JWT 라이브러리)**
   - HMAC 커스텀 장점: 구현 단순, 의존성 적음, 학습 목적에 유리
   - HMAC 커스텀 단점: 표준 클레임/만료/키회전/검증옵션을 직접 구현·검증해야 함
   - JWT 라이브러리 장점: 표준 클레임(`exp`,`iat`,`iss`), 검증 옵션, 생태계/호환성 우수
   - JWT 라이브러리 단점: 의존성 증가, 초기 학습 비용
   - **결정**: 현재는 HMAC 유지(개발 속도), 단 `JWT 전환 체크포인트`를 잔여업무에 등록
3. **로그아웃 정책 비교 결론 (무상태 success vs 서버 블랙리스트)**
   - 무상태(success) 장점: 서버 단순, 저장소 불필요, 확장 쉬움
   - 무상태(success) 단점: 발급된 토큰 즉시 무효화 불가(만료까지 유효)
   - 서버 블랙리스트 장점: 로그아웃/탈취 대응 시 즉시 차단 가능
   - 서버 블랙리스트 단점: 저장소/만료청소/분산동기화 필요
   - **블랙리스트 의미**: "더 이상 신뢰하지 않는 토큰 목록"을 서버가 보관하고 검증 시 거부하는 방식
   - **결정**: 현재 서버측 블랙리스트(메모리 denylist) 유지, 추후 Redis/DB 영속화
